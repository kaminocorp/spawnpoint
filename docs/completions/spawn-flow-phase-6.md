# Phase 6 Completion — Spawn Flow: Fleet page at `/fleet`

**Plan:** `docs/executing/spawn-flow.md` §Phase 6
**Roadmap:** `docs/plans/post-0.2.6-roadmap.md` §M4
**Date:** 2026-04-26
**Status:** complete; checkpoint green (`pnpm -C frontend type-check && pnpm -C frontend lint && pnpm -C frontend build` all clean; `/fleet` flips from M1's `<ComingSoon>` stub to a real listing with the four-state pattern; the M4 user-visible loop is now closed in code — Phase 7 is the runtime walkthrough)

This phase closes the **second half of the user-facing spawn loop**. Phase 5 made the Deploy modal land the user on `/fleet` via `router.push("/fleet")`; pre-Phase-6 that destination was M1's static `<ComingSoon>` placeholder. Phase 6 replaces it with the discriminated-union page (`loading | empty | ready | error` per plan decision 31), the 3-second polling lifecycle (decision 41), the status-badge color taxonomy (decision 31), the per-row Stop / Destroy `AlertDialog` confirmations (decision 42), and the external Logs link (decision 33). **First time the FE round-trips on a server-driven state convergence** — the table re-renders as `pending → running` flips in the BE, with no streaming, no SSE, just a refetch-on-interval that idles to zero cost once every row settles.

This phase also introduces **the second modal in the codebase** (`AlertDialog`, distinct from Phase 5's `Dialog`) and **the first read+write surface where reads are cheap and writes are confirmed**. The Stop and Destroy actions sit one click away from a Fly-app destruction; the AlertDialog wrapper is the FE-side enforcement of plan decision 42's "destructive enough to warrant confirmation." Optimistic UI was deliberately *not* picked (decision 43) — the 1–3s wait for `scale 0` / `apps destroy` is acceptable, and rolling back optimistic state on RPC failure is more surface area than v1 needs.

## What this phase shipped

### The four-state page

```tsx
type State =
  | { kind: "loading" }
  | { kind: "ready"; instances: AgentInstance[] }
  | { kind: "empty" }
  | { kind: "error"; message: string };
```

Identical shape to the M2 catalog page (decision 31's "mirrors dashboard's four-state shape from 0.3.0" — the catalog page borrowed it, fleet page borrows it again). The four-state pattern is **the established v1 frontend idiom for "RPC-backed list view that can also be empty or fail"** — three files now use it (catalog, fleet, and `loading|ready|not-provisioned|error` on onboarding).

`empty` and `loading` are both rendered as Cards / Skeletons centered on the page; `ready` renders the table; `error` renders a Card with the ConnectError message and a hopeful "this might recover on its own" line that's literally true once the polling effect kicks back in next render — see Decisions §2 for why error doesn't auto-retry.

### The two-`useEffect` lifecycle

```tsx
// Effect 1: initial fetch on mount, no dependencies-on-state.
useEffect(() => {
  let cancelled = false;
  void (async () => {
    await fetchInstances();
    if (cancelled) return;
  })();
  return () => { cancelled = true; };
}, [fetchInstances]);

// Effect 2: conditional polling — runs only while non-terminal rows exist.
useEffect(() => {
  if (state.kind !== "ready") return;
  if (state.instances.every((i) => isTerminal(i.status))) return;
  const id = setInterval(() => { void fetchInstances(); }, POLL_MS);
  return () => clearInterval(id);
}, [state, fetchInstances]);
```

**Two effects, not one.** Effect 1 fires once on mount. Effect 2 watches `state` and conditionally starts/stops the interval. When the state transitions (e.g. `ready` with one pending row → `ready` with all-terminal rows), Effect 2's cleanup fires and the interval clears. **The polling cost is exactly zero once every row has settled** — no `setInterval` is alive, no fetch is in flight. This matches plan decision 41's "the hot loop only runs while at least one agent is converging."

The `fetchInstances` callback is `useCallback`-wrapped with no dependencies; both effects depend on its stable identity. **The dependency-array trick that makes this work**: `setState` is a setter (stable identity guaranteed by React); `createApiClient()` is constructed inside the callback so it's not closed over from outside; `setState` calls take only the new state value. The result is one fetch implementation, two effects that consume it, and zero re-fetch storms from identity churn.

### Status taxonomy (decision 31)

| Status | Badge variant | Label | Extra className | Why |
|---|---|---|---|---|
| `pending` | `secondary` | "Pending" | — | Neutral. The default secondary (muted gray) reads as "waiting" without claiming success or failure. |
| `running` | `secondary` | "Running" | `bg-emerald-500/15 text-emerald-700 dark:text-emerald-400` | Green per decision 31. shadcn doesn't ship a "success" variant; the className override layers emerald onto secondary. |
| `stopped` | `outline` | "Stopped" | `text-muted-foreground` | Gray per decision 31. Outline + muted-foreground reads as "intentionally idle." |
| `failed` | `destructive` | "Failed" | — | Destructive per decision 31. shadcn's destructive variant is the right red shape out of the box. |
| `destroyed` | `outline` | "Destroyed" | `text-muted-foreground/70 line-through` | Muted-destructive per decision 31. Line-through is **a distinct visual signal** for "this row exists for audit, not management" — the cell still renders the name, but the strikethrough reads as past-tense at a glance. |

The CSS-class layering (`secondary` + emerald override for running) is the right cost trade. Adding a green Badge variant to the shadcn primitive would be **the wrong scope** — the project doesn't have a generalized "success" semantic to support. One ad-hoc override at the consumer site beats touching the design system primitive.

### Per-row actions (decision 42)

Three conditionally-visible buttons in a right-aligned flex row:

| Button | Visible when | Variant | Confirmation | Effect |
|---|---|---|---|---|
| Logs | `logsUrl !== ""` | `ghost` (with `ExternalLinkIcon`) | none — read-only | Opens the Fly dashboard URL in a new tab (`target="_blank" rel="noopener noreferrer"`) |
| Stop | `status === "running"` | `ghost` (with `OctagonIcon`) | `AlertDialog` with "Stop {name}?" copy | `api.agents.stopAgentInstance({ id })`; toast on success; row refetches |
| Destroy | `status !== "destroyed"` | `ghost` + `text-destructive` (with `TrashIcon`) | `AlertDialog` with "Destroy {name}?" copy + destructive action button | `api.agents.destroyAgentInstance({ id })`; toast on success; row refetches |

Logs is **deliberately not gated on `status`** — even a `failed` or `stopped` row may have a `logsUrl` (the agent crashed; the user wants the logs). The visibility predicate is exactly `logsUrl !== ""`, matching the empty-string sentinel from the proto wire shape (which the BE service computes only after `deploy_external_ref` is set per spawn-flow plan decision 33).

Stop's "this can be destroyed and re-spawned, but not started again in v1" copy is **the user-facing surface of plan decision 23's start-after-stop deferral**. Today's UX is honest about the limitation; v2 unblocks restart with a "rotate keys?" flow per the plan's deferred decisions.

Destroy's "the row stays for audit" copy is the user-facing surface of plan decision 24's soft-delete semantics. The user sees the destroyed row in a muted line-through state for the rest of the session — **that's the audit trail being legible**, not just present.

### Confirmation flow

`AgentRowActions` owns one `pending: "stop" | "destroy" | null` state. Click Stop → `setPending("stop")` → AlertDialog opens with stop-specific title/description/button; Cancel or backdrop click → `setPending(null)` (gated on `!submitting`). Click Confirm → fires the corresponding RPC, on success → `setPending(null)` + `onChanged()` (the parent's refetch callback) + toast.

The `onChanged()` callback is the **parent-driven refresh hook** — `AgentRowActions` doesn't manage the page state, it just signals "the row may have changed; please re-fetch if you care." The fleet page passes `fetchInstances` as `onChanged`, which causes the `state` to update, which causes Effect 2 to re-evaluate the polling-lifecycle predicate. **Decoupling refresh from the action component** keeps the actions reusable (the same component would work in a "single agent detail" view if Phase 7+ adds one).

### Provider + timestamp formatting

Two small render helpers in `page.tsx`:

- `providerLabel(p: ModelProvider)` — switch over the proto enum ints, returns lowercase strings (`anthropic` / `openai` / `openrouter` / `—` for `MODEL_PROVIDER_UNSPECIFIED`). Lowercase because the cell renders mono — `anthropic` reads more naturally there than `Anthropic`.
- `formatCreated(rfc3339: string)` — empty-string sentinel → `—`; invalid date → fall back to the raw string (defensive); valid date → `Intl.DateTimeFormat` with `month: "short", day: "numeric", hour: "2-digit", minute: "2-digit"`. Uses the user's locale by default; no library dependency. **`Number.isNaN(d.getTime())` is the canonical "this Date didn't parse" guard** — `instanceof Date` doesn't catch invalid dates, only non-Date values.

## Decisions made this phase

### 1. Two `useEffect`s, not one combined effect

Considered (and rejected): a single effect that fetches once and starts a conditional `setInterval` based on the response.

**Rejected because:**
- The single-effect shape would re-create the interval on every successful fetch (because the response is the dependency), causing churn.
- Suppressing churn requires a `useRef` to hold the interval ID, plus manual mount/unmount handling, plus a separate predicate for "should the interval be running right now?" — three pieces of state to coordinate vs React's already-coordinated effect lifecycle.
- The `[state, fetchInstances]` dependency array on Effect 2 is **the right shape**: when `state` changes (from `loading` to `ready`, or from "all pending" to "all settled"), React tears down the previous interval and decides whether to set up a new one.

The two-effect split is **idiomatic React-with-effects** for "fetch once, then conditionally poll." The cost is one extra `useEffect`; the benefit is that the polling lifecycle is fully declarative — there's no imperative "start polling" / "stop polling" state machine to debug.

### 2. No auto-retry on `error`; rely on next user-driven mount

The `error` state renders a Card and stops there. **No retry button, no auto-retry timer.** The reasoning:

- **The user knows when to come back.** A network blip during list-fetch is the typical cause; the user reading "Couldn't load the fleet" understands they can refresh the tab. Adding a retry button suggests the page is in a recoverable state at all times, which it isn't.
- **An auto-retry timer would mask BE-side outages.** If the BE is down, polling-on-error would generate a steady stream of failed RPCs; the user-driven retry is **the natural backpressure** that prevents the FE from hammering a sick BE.
- **The error message itself is informative.** ConnectError surfaces the BE-side sentinel via the message string (post-Phase-4's `agentsErrToConnect` redaction layer); the user can see "internal error" vs "unauthenticated" and act accordingly.

If Phase 8 hardening reveals a class of transient errors that should auto-retry (e.g. a 502 between Vercel and Fly during deploy), that's an additive change. **The default-no-retry shape leaves the door open without committing to logic that's hard to right-size at v1.**

### 3. `pending: "stop" | "destroy" | null` over two booleans

Considered: `const [stopOpen, setStopOpen] = useState(false); const [destroyOpen, setDestroyOpen] = useState(false);` and one AlertDialog instance per action.

**Rejected because:**
- Two AlertDialog instances per row × N rows = 2N modals in the DOM; tagged-null with one modal per row is N.
- The cross-state invariant "at most one of (stopOpen, destroyOpen) is true" lives implicitly in callsite assumptions; tagged-null encodes it at the type level.
- The Confirm button's logic branches once on `pending === "stop"` vs `pending === "destroy"` — the same place either way; two-boolean form would scatter the branch across two confirm handlers.

**Picked:** one `pending` state, one `AlertDialog`, one confirm function with a `pending === "stop"` branch. The `AlertDialog`'s `open={pending !== null}` derived predicate keeps the Dialog state and the action state in lockstep without a useEffect.

### 4. `onChanged: () => void` callback over event-bus / ref / context

`AgentRowActions` doesn't know about `fetchInstances` — it knows it has an `onChanged` callback to fire after a successful action. The fleet page passes `fetchInstances` as `onChanged` for every row; same callback identity across rows.

**Why this shape and not, say, a context for "the fleet refresh function":**

- One row, one action, one parent — context would be over-engineered for a unidirectional flow this short.
- Passing the callback as a prop **makes the data flow legible at the JSX site** — the reader of `<AgentRowActions onChanged={fetchInstances} />` sees the wiring without chasing a `useFleetRefresh` hook.
- If Phase 7+ adds an "agent detail page" that hosts the same actions component, the callback prop slots in unchanged; a context-based design would require provider plumbing per consumer.

This is the same shape as M1's onboarding submit handlers: explicit callback prop, no implicit data layer.

### 5. AlertDialog Cancel uses base-ui's Close primitive (no manual onClick)

`AlertDialogCancel` is rendered as a `<Button variant="outline">` wrapping base-ui's `AlertDialog.Close` (the shadcn wrapper does this internally). Clicking it closes the dialog via the primitive's own state machine — **no manual `onClick={() => setPending(null)}` needed**. Confirm, however, *does* need a manual onClick (`onClick={confirm}`) because Confirm is *not* a Close — it fires the RPC, then closes via `setPending(null)` only on success.

The asymmetry (Cancel = Close, Confirm = action) is **a deliberate property of the AlertDialog primitive** — Cancel's purpose is "user changed their mind, close it"; Confirm's purpose is "do the thing, possibly stay open if it fails." Today's confirm path closes on success; on failure the toast fires and the modal stays open so the user can retry — this is a deliberate divergence from Phase 5's deploy modal (which closes the modal then redirects, intentionally taking the user away from the form on failure too via the redirect).

- **Tab visibility-aware polling.** Plan Q5 + decision 41 — pause polling when the tab is backgrounded. Polish, not v1; surfaces if the demo runs with `/fleet` open in a hidden tab and the user checks back.
- **Sortable columns.** The plan explicitly skips this (§5 anti-scope-creep "search/filter UI"); flagging because once the fleet has 10+ rows the lack of sort is felt. **The `(org_id, created_at DESC)` index from Phase 1 is the substrate**; the FE is the missing half.

- **Long-running fetch state.** Today the second-and-subsequent fetches don't surface a "refreshing" indicator; the user only sees the result render. A subtle indicator (e.g. a top-of-page spinner) would matter for slower BE round-trips. v1 BE is fast enough this isn't visible.
- **Row-level loading states for Stop / Destroy.** Today the AlertDialog's button transitions to `Stopping…` / `Destroying…` — but the table row itself doesn't dim or change. If a user dismisses the dialog mid-RPC (which the gate prevents), the row would be temporarily in an indeterminate state. **The dialog gate is the load-bearing guard**; row-level dimming would be additive polish.
- **Error toast deduplication.** If the same error fires repeatedly via the polling loop (e.g. transient network failure during the polling window), the user sees N toasts. `sonner` has a `dedupe`-ish behavior via `toast.id`; flag for if/when it becomes visible.
- **Skeleton matching the table layout.** Today `LoadingTable` renders three full-width skeletons — not column-shaped. The table renders on first successful fetch; the visual transition from skeleton → table is mildly jarring because the column widths shift. Cosmetic; flag for Phase 8.

**Decision 32's stale-pending sweep is unchanged** — Phase 4 wired it at boot per the plan; Phase 6's polling lifecycle is the FE counterpart that *displays* the sweep's result (a formerly-pending row that's now `failed`).
