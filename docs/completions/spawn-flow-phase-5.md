# Phase 5 Completion — Spawn Flow: Frontend deploy modal at `/agents`

**Plan:** `docs/executing/spawn-flow.md` §Phase 5

**Date:** 2026-04-26
**Status:** complete; checkpoint green (`pnpm -C frontend type-check && pnpm -C frontend lint && pnpm -C frontend build` all clean; `/agents` page now exposes two active CTAs per Hermes card; deploy modal is the first user-reachable code path that hits `SpawnAgent` / `SpawnNAgents` over the wire that Phase 4 made callable)

This phase closes the **first half of the user-facing spawn loop**: from M2's catalog page, the user can now click a real Deploy button, fill a real form, and submit — the form hits Phase 4's six-handler surface for the first time over a live wire. The other half (post-spawn fleet view + status convergence) is Phase 6's territory; for Phase 5 the success path lands the user on `/fleet` (still M1's `<ComingSoon>` stub today) via `router.push("/fleet")` per plan decision 38. **First time the FE has a write-shaped RPC that isn't tied to onboarding** — `users.updateCurrentUserName` and `organizations.updateOrganizationName` from M1 are the only prior writes; everything else (`getCurrentUser`, `listAgentTemplates`) is read-only. This phase establishes the FE pattern for "write-then-redirect-and-poll" that Phase 6 will round-trip on.

This is also the **first dialog (modal) in the codebase** — the M1 wizard is a full-page route, the M2 catalog is grid-only, the dashboard cards are read-only. Phase 5 introduces the `Dialog` shadcn component (`@base-ui/react`-backed) and the in-modal form pattern. The form pattern itself is unchanged from M1's `react-hook-form` + `zodResolver` + raw `<Label>` + `<Input>` shape — no `<Form>` wrapper from shadcn, consistent with the M1 onboarding precedent (one fewer layer of indirection per field, and `useWatch` is the React-Compiler-friendly way to read controlled values).

## What this phase shipped

### The deploy modal — one component, two form variants

The wrapper component `DeployModal` is small (~25 LOC of layout + state hoisting); the two form variants share zero React state but share a `FormProps` shape (`template`, `submitting`, `setSubmitting`, `onClose`). The `submitting` state is **lifted to the wrapper** so the `Dialog`'s `onOpenChange` callback can reject outside-click and ESC dismissals while an RPC is in flight (plan Q4):

```tsx
function handleOpenChange(next: boolean) {
  if (submitting && !next) return;   // refuse close-attempts during submit
  onOpenChange(next);
}
```

The "Cancel" button is independently `disabled={submitting}`, and the close-X (`showCloseButton`) flips off during submit. **Three independent close paths, all gated on the same submitting flag** — an in-flight RPC cannot be dismissed by any user action. The submit button's own disabled state is the fourth gate (no double-submit).

### Form + validation shape

Two zod schemas — `oneSchema` and `manySchema` — share three field defs via spread (`provider`, `modelName`, `apiKey`); `oneSchema` adds `name`, `manySchema` adds `namePrefix` + `count`. Field-level constraints come straight from plan decision 13 + 14 + 26:

| Field | Constraint | Source |
|---|---|---|
| `name` | `trim().min(1).max(80)` | decision 26's "1–80 chars" + the BE-side check at `agents.Service.Spawn` |
| `namePrefix` | `trim().min(1).max(60)` | decision 14's name-prefix bounds (60 leaves headroom for the `-NN` suffix decision 28 adds) |
| `count` | `int().min(1).max(10)` | decision 14's "capped at 10" — re-validated server-side via `ErrSpawnLimit` (decision 25) |
| `provider` | enum `anthropic / openai / openrouter` | decision 13's three-element provider list; matches proto's `ModelProvider` excluding `UNSPECIFIED` |
| `modelName` | `trim().min(1).max(200)` | decision 26's "1–200 chars" |
| `apiKey` | `min(1)` | decision 26's "non-empty"; **deliberately no `.trim()`** — provider keys can theoretically have leading whitespace and trimming would silently mutate the value before forwarding |

The provider field uses `useWatch({ control, name: "provider" })` rather than `form.watch("provider")` to surface a memoizable value (see Decisions §3); the `count` field uses `register("count", { valueAsNumber: true })` so the HTML `type="number"` string becomes a number before zod sees it (avoids the `z.coerce.number()` type-drift that broke the resolver's input/output types when first attempted — see "Decisions made this phase" §2).

### The API key field — show toggle + audit copy

The API key `<Input>` is `type="password"` by default with a one-button toggle (Lucide `EyeIcon` ↔ `EyeOffIcon`) that flips it to `type="text"`. Inline helper copy under the field reads:

> Forwarded once to the agent's secret store. Never written to Corellia's database.

This is the only inline copy in the form that talks about **what happens to the value**, not just what the field expects. Plan decision 7 frames the API key as the single most security-sensitive value in the request; the inline copy is the user-facing surface of that invariant. Plan decision 39 specifies "no localStorage" — the form value lives in `react-hook-form`'s in-component state only, so a refreshed tab loses the in-flight value (acceptable; spawn is one-shot).

### Two-button card footer

`AgentTemplateCard`'s footer is now:

```tsx
<CardFooter className="justify-end gap-2">
  <Button variant="outline" onClick={() => openWith("many")}>Deploy 5</Button>
```

`Deploy` is the default variant (primary action — single-spawn is the typical user flow); `Deploy 5` is `outline` (secondary — the "demo moment" affordance per blueprint §10's last paragraph). Both open the same `<DeployModal>` instance; the only difference is the `mode` prop. **One modal, two buttons** — exactly plan decision 40's shape. The `gap-2` between buttons mirrors the `CardFooter` spacing rhythm from the rest of the catalog grid.

## Decisions made this phase

### 1. No `lib/api/agents.ts` wrapper file — generated client *is* the wrapper

Plan §Phase 5 task 3 said: *"`frontend/src/lib/api/agents.ts` — extend M2's catalog client. Add wrappers for `spawnAgent`, `spawnNAgents`, `listAgentInstances`, `getAgentInstance`, `stopAgentInstance`, `destroyAgentInstance`. Each calls the generated Connect TS client with the auth header pattern from M2."*

**This file does not exist in M2** — and never did. The M2 catalog page (`frontend/src/app/(app)/agents/page.tsx` line 35) calls `api.agents.listAgentTemplates({})` directly via the `createApiClient()` return value (`frontend/src/lib/api/client.ts` lines 24–28), which already exposes `agents` as `createConnectClient(AgentsService, transport)`. The auth-header pattern is wired once at the transport level (`client.ts` lines 12–22), not per RPC. **A hand-written wrapper layer with no extra args, no normalization, and no caching would just duplicate every generated method name.** That's the opposite of "abstractions earn their keep" — and CLAUDE.md's "Don't add features beyond what the task requires" rule flags it directly.

**Phase 5 skips the wrapper file.** The deploy modal calls `api.agents.spawnAgent({...})` and `api.agents.spawnNAgents({...})` directly via `createApiClient()`, same pattern as M2's `api.agents.listAgentTemplates({})`. Phase 6's fleet page will follow the same shape for `listAgentInstances` / `getAgentInstance` / `stopAgentInstance` / `destroyAgentInstance`.

This is plan-as-built drift in the same vein as Phase 4 §1 (`OutOfRange` vs `FailedPrecondition`): the plan author's grep was correct at draft time but the codebase shape diverged in the meantime. Both are one-line corrections to the plan's prescribed file list, neither is a structural change.

### 2. `z.number()` + `valueAsNumber: true`, not `z.coerce.number()`

First attempt used `z.coerce.number().int().min(1).max(10)` for the `count` field. This produces a zod schema whose **input type** is `unknown` (anything coerce-able to a number) but whose **output type** is `number`. `react-hook-form`'s `Resolver<TFieldValues>` type wants `TFieldValues` to align with the form's value type — but the form's `defaultValues.count = 5` is `number`, while the resolver expects `unknown`. TypeScript flagged the divergence as `Type 'unknown' is not assignable to type 'number'` at the `useForm` call site.

**Fix:** drop `z.coerce` (use `z.number()` directly) and add `valueAsNumber: true` to the `count` field's `register()` call. This shifts the string-to-number coercion from zod (input shape) to react-hook-form's `register()` (input shape — handled at the field-binding layer before the resolver sees it). The schema's input and output types align (`number` → `number`), the `Resolver` type is happy, and the runtime behavior is identical: the HTML `type="number"` input emits a string, RHF coerces to number on read, zod validates the number.

**Why this matters going forward:** any future numeric form field should follow the same pattern (`z.number()` + `valueAsNumber: true`) rather than `z.coerce.number()`. The latter looks tempting because it's "schema-only" but it pushes type complexity into the resolver layer. The former is the canonical shape for HTML number inputs.

### 3. `useWatch`, not `form.watch()` — React Compiler memoization

The first lint pass surfaced two warnings:

> React Hook Form's `useForm()` API returns a `watch()` function which cannot be memoized safely. … this component will not be memoized by the React Compiler.

`form.watch("provider")` returns the current value but the function reference changes per render, defeating React Compiler's memoization heuristics. The fix is to use the standalone `useWatch({ control: form.control, name: "provider" })` hook — which subscribes via `control` (a stable reference) and returns a memoizable value.

**Two extracted lines per form** — one `const provider = useWatch(...)` and one `const count = useWatch(...)` (the `DeployManyForm` only). Three behavioral consequences:

1. The component is now eligible for React Compiler memoization (warnings cleared).
2. The values are stable references across renders for downstream prop comparisons (matters if `<ProviderField>` ever gets `React.memo`'d).
3. The data-flow is clearer at the function header: a glance at the top of `DeployOneForm` sees "this form watches `provider`" — `form.watch("provider")` inline at the JSX site doesn't surface the same way.

This is the established React Compiler-friendly shape going forward. Future forms with watched fields should use `useWatch`, not `form.watch()`.

### 4. Two parallel forms (one + many), not one discriminated form

Considered (and rejected): a single `useForm` with a discriminated-union schema (`z.discriminatedUnion("mode", [oneSchema, manySchema])`) and conditional field rendering based on `mode`.

**Rejected because:**
- The two schemas share three fields, not one — `name` vs `namePrefix + count` is the *only* difference, but it's a meaningful one (the `defaultValues` shape differs, the `register()` calls differ, the submit handler's call to the API differs).
- `useForm`'s `TFieldValues` generic is invariant — switching modes would require type assertions on every `register()` call, defeating the point of the resolver type-check.
- React's `key={mode}` on the wrapper component already gives free remount-on-mode-change, which means no shared form state can leak across modes anyway.
- The duplicated JSX is ~30 LOC of field rendering, of which ~25 LOC is shared (extracted into `Field`, `ProviderField`, `ApiKeyField`).

**Picked:** `DeployOneForm` and `DeployManyForm` as separate components, each with its own `useForm`. Modal wrapper switches on `mode === "one"` with a React `key` prop. The form value types are independent at the type level, the submit logic is independent at the runtime level, and the shared field rendering is extracted into three small subcomponents (`Field`, `ProviderField`, `ApiKeyField`).

### 5. `submitting` state lifted to modal wrapper, not local to forms

First attempt had `submitting` state local to each form component. This left the `Dialog`'s `onOpenChange` (controlled at the wrapper level) unable to consult the value, so outside-click during submit would close the modal and abandon the in-flight RPC.

**Fix:** lift `submitting` + `setSubmitting` to the `DeployModal` wrapper, pass both down to each form via `FormProps`. The wrapper's `handleOpenChange` early-returns `if (submitting && !next)` — refusing all close attempts (outside-click, ESC, X-button — though the X is also gated via `showCloseButton={!submitting}`).

Plan Q4 framed this as "modal close-on-outside-click — disable while submitting; default to disable on submitting." The implementation matches: closes are **disabled across all paths** during submit, not just outside-click.

## Known pending work

**Phase 6** (fleet page) — the post-redirect destination. M4's loop is open at `/fleet`'s door: the deploy modal's `router.push("/fleet")` lands on the M1 `<ComingSoon>` stub today. Phase 6 replaces it with the discriminated-union page (`loading | empty | ready | error`), the 3s polling lifecycle (decision 41), the status-badge component, the Stop / Destroy `AlertDialog` confirmations, and the per-row Logs link. **The deploy modal is structurally ready for Phase 6's arrival** — the toast-then-redirect shape doesn't change; only what's *at* `/fleet` does.

**Phase 7** (integration smoke test) — first end-to-end click-through against a live BE + a real Fly account + a real provider key. Phase 5's build output guarantees the FE compiles and the wire surface aligns with Phase 4's handlers; the unknowns at Phase 7 are runtime-only (the 30s `pending → running` convergence, the redaction layer's behavior under a bad provider key, the `Deploy 5` semaphore feeling parallel rather than serial in the demo).

**Phase 8 hardening pre-flagged candidates for the modal:**

- **`shadcn` Form wrapper.** The Dialog's form uses raw `<Label>` + `<Input>` per the M1 onboarding precedent. If Phase 6's fleet page introduces shadcn's `<Form>` for filter UI (post-v1 anyway), the deploy modal could be migrated for consistency. Today neither precedent calls for it; the raw shape is one fewer layer of indirection per field.
- **Form value persistence across modal open/close.** Today closing the modal mid-fill discards all form state — re-opening starts fresh (`useForm` re-initializes on remount). For a hackathon-scoped one-shot flow this is acceptable; for v1.5 polish a "drafts" mechanism could persist field values to component-scope state above the modal.
- **`Deploy 5` count default = 5 vs `Deploy 1` flow.** The default `count: 5` matches the demo's "fan out 5" framing but means a user who mis-clicked `Deploy 5` and meant `Deploy` would have to either change the count to 1 or close + click `Deploy` instead. Cosmetic; reconsider after Phase 7's smoke.
- **Submit-then-redirect race.** `setSubmitting(false) → onClose() → router.push("/fleet")` is sequential, but the Next.js navigation is async — there's a brief window where the modal is closed and the page hasn't re-routed. The current order avoids the modal flashing closed-then-redirected; if Phase 6 surfaces the BE-side row insert as a render-blocking SSR fetch, this race may surface as a "no row visible yet" flicker.
- **API key field `paste` event.** Some users paste keys with a leading space (UX habit from copy-paste). `apiKey: z.string().min(1)` deliberately doesn't `.trim()` — but if a user keeps hitting "Required" because of leading whitespace, the right fix is at the input layer (a paste handler that trims), not the schema. Flag for Phase 7's smoke run.
- **Provider list provider-locked model lists.** Today `modelName` is a free-text input with placeholder `claude-opus-4-7`. A `<Select>` of provider-specific model names would catch typos earlier (decision 26's "model name not validated against a list" sticks for v1; Phase 8 hardening could surface a curated provider→models map without re-validating server-side).
- **`Deploy 5` button state when count would be 0.** Today the count default is 5; if a user fans out and types 0, the form rejects via zod's `min(1)`. The submit button label reads "Deploy 0" briefly until validation fires. Cosmetic; not a defect.

**Decision 32's stale-pending sweep is unchanged** — Phase 4 wired it at boot per the plan, Phase 5 doesn't touch it.

**No new sentinels or Connect codes this phase.** The eight M4 sentinels Phase 4 wired (`ErrInvalidName` through `ErrTargetUnavailable`) flow through to the modal via `ConnectError.from(e)` → `err.message` → `toast.error(...)`. **First time these sentinels are visible to a real user** — Phase 7's smoke will confirm the redaction layer (`ErrFlyAPI` → `Unavailable` with generic message) reads correctly to the FE.
