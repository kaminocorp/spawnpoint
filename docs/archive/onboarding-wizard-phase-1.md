## Why each call

### Why `<Toaster />` lives in the root layout, not the `(app)/` chrome layout (Phase 3)
A Toaster portal renders globally — moving it into the chrome layout would mean toasts can't fire on the sign-in page or the standalone wizard route (Phase 2). The wizard's success/error feedback needs toasts to work, so the Toaster has to live above the route group boundary. Root layout is the only place it works for *all* routes.

### Why `richColors closeButton` rather than defaults
- `richColors` enables sonner's per-variant tinting (`success` is green, `error` is red, etc.) — visually clearer than the monochrome default and matches the colour-aware `cn-toast` class shadcn's `sonner.tsx` declares.
- `closeButton` adds an explicit dismiss affordance. Toasts auto-dismiss after a few seconds, but error toasts often need to be re-read; the X button covers that case without keyboard trickery.

### Why `useSyncExternalStore` instead of suppressing the lint rule
The shadcn-shipped `use-mobile.ts` calls `setIsMobile` synchronously inside a `useEffect`, which trips the project's strict `react-hooks/set-state-in-effect` ESLint rule (the rule fires on React 19's stricter cascading-render guidance). Two options:

1. Inline `// eslint-disable-next-line` on the offending line.
2. Refactor to `useSyncExternalStore` — React's canonical primitive for subscribing to external state sources (matchMedia is exactly that).

Picked (2). Reasons:
- **Correctness.** The original shape has a known SSR-hydration issue: `useState<boolean | undefined>(undefined)` means the server renders `false` (the `!!undefined` coercion in the return), the client mounts and runs the effect, then re-renders with the actual value. `useSyncExternalStore` with `getServerSnapshot` returning `false` makes the contract explicit and the hydration consistent.
- **One-time cost.** ~21 LOC, structurally simpler than the original (no `useState`, no `useEffect`, no cleanup function — `useSyncExternalStore` handles all three).
- **Vendor drift risk is bounded.** If shadcn updates `sidebar.tsx` later in a way that depends on the old hook's signature, the API is unchanged (`useIsMobile(): boolean`); only the implementation differs. We can re-`shadcn add use-mobile` at any time and re-apply this rewrite — the diff is small and the rationale is in this doc.
- **Suppressing the rule is the wrong precedent.** The project enforces `react-hooks/set-state-in-effect` deliberately; carving out an exemption for one vendor file invites copy-paste exemptions later.

### Why no `ThemeProvider` despite `next-themes` being installed
The shadcn `sonner.tsx` wrapper calls `useTheme()` from `next-themes`. Without a `ThemeProvider`, that hook returns `{ theme: undefined }`, which the wrapper handles via the `theme = "system"` default destructure — no crash, the toaster just doesn't follow theme overrides.

The plan's decision #15 explicitly defers dark mode to a later polish plan. Mounting `ThemeProvider` here would be scope creep: it implies a toggle, a hydration-mismatch guard (`suppressHydrationWarning` on `<html>`), and a system-preference media query — a separate plan's worth of decisions. Toasts working in light mode only is acceptable for M1.

### Why metadata changed to `"Control plane for AI agents."` rather than something longer
Vision.md's opening sentence is "Corellia is a centralized control plane for AI agents — a single platform to spawn, deploy, govern, and manage agents at scale, across any model, any provider, and any agent framework." The first ten words are the load-bearing summary; the rest is differentiator copy. Browser-tab descriptions and link-preview cards typically truncate at ~150 chars, so the short form is the right target.

## Deviations from the written plan

### 1. `form.tsx` not added — shadcn registry doesn't ship one for the `base-nova` style

The plan listed `form` in the `shadcn add` command. The CLI accepted the argument, checked the registry, and exited cleanly without creating the file (no error, no warning — just a silent skip). Verified with a follow-up explicit `shadcn add form` call: same silent success, no file created.

**Why this is OK.** The shadcn `Form` primitive is a thin wrapper around `react-hook-form`'s `FormProvider` + `Controller` + `useFormContext` plus shadcn `Label` and a custom `FormMessage` for inline errors. It's a convenience, not a requirement. The existing `frontend/src/app/sign-in/page.tsx` already uses raw `<Input>` + `<Label>` + `useState` + manual error rendering — Phase 2's wizard will follow the same pattern (with `useForm` + `zodResolver` instead of `useState`, since validation is more involved).

**Concrete adjustment to Phase 2's plan.** The wizard will use:
```tsx
const form = useForm<Schema>({ resolver: zodResolver(schema) });
// ...
<form onSubmit={form.handleSubmit(onSubmit)}>
  <Label htmlFor="name">…</Label>
  <Input id="name" {...form.register("name")} />
  {form.formState.errors.name && (
    <p className="text-sm text-destructive">{form.formState.errors.name.message}</p>
  )}
</form>
```

This is mechanically identical to using shadcn's `Form` primitive, just without the named `FormField`/`FormItem`/`FormLabel`/`FormMessage` wrappers. Ergonomically a touch noisier; semantically equivalent. Worth one bullet in the Phase 2 completion doc to flag the inconsistency, but not worth blocking on.

### 2. `use-mobile.ts` rewritten

Already covered in §"Why each call" above. Listed here so the deviation is searchable.

### 3. Eight files added, not six

Not strictly a deviation from intent — the plan said "install missing shadcn primitives" — but worth noting that `sidebar`'s implementation depends on `sheet` and `tooltip`, both of which the CLI pulled in transitively. The `use-mobile` hook also came along for `sidebar`'s mobile-collapse logic. This is canonical shadcn behaviour, not a surprise; flagged for completeness so future reviewers don't wonder why `sheet.tsx` exists when nobody asked for it.

## Findings (out of Phase 1 scope, surfaced for follow-up)

### Pre-existing build failure: `/sign-in` prerender throws on `pnpm build`

`pnpm -C frontend build` fails during static-export generation:

```
Error occurred prerendering page "/sign-in".
Error: @supabase/ssr: Your project's URL and API key are required to create a Supabase client!
```

**Root cause analysis.**
- `app/sign-in/page.tsx` is a `"use client"` component but Next.js still runs it through prerender to produce a static HTML payload for the route. During prerender, Next 16's Turbopack build worker reads `frontend/.env.local` (the build banner confirms `Environments: .env.local`), but the value of `NEXT_PUBLIC_SUPABASE_URL` is either missing or empty.
- `lib/supabase/client.ts` calls `createBrowserClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, …)` — the `!` non-null assertion hides that the function throws if the value is empty.
- `app/sign-in/page.tsx` calls `createClient()` *inside the component body* (above the JSX, top of the function). That means the call runs at every render, including prerender — which is why the throw escapes to the build.

**Confirmed pre-existing.** I `git stash`'d all Phase 1 changes and re-ran `pnpm build` on clean `master`. Same error, same trace. Phase 1 didn't introduce or worsen this; it was broken on the commit before mine too.

**Why it doesn't block Phase 1.** The plan's pre-work checklist listed only `type-check` + `lint` as the "builds clean today" gate (both green, here and on master). Production build is in Phase 6's acceptance, not Phase 1's. The fix belongs to either (a) Phase 6, or (b) a separate hardening task that pre-dates this milestone.

**Recommended fix path** (for whoever picks up the build-clean follow-up — not me, this milestone):
- Move `createClient()` into the `onSubmit` handler so it only runs at user-interaction time, never at prerender.
- *Or* mark the route `export const dynamic = "force-dynamic"` to skip prerender entirely.
- *Or* add SSR-safe guards in `lib/supabase/client.ts` that throw a helpful error at *runtime* with debug context, not at build time on missing env.

The first is cleanest — `createClient()` is cheap to construct on demand, and the `supabase.auth.signInWithPassword` call is the only place the page actually needs it.
