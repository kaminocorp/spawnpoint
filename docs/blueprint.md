# Corellia — Technical Blueprint

The technical counterpart to `vision.md`. Where `vision.md` captures *what* Corellia
is and *why*, this document captures the concrete engineering decisions that
implement that vision. It is the single source of truth for architecture,
data model, scope, and codebase norms.

When architectural decisions change, update this document. When it conflicts with
any other document in the repo, this document wins.

Companion docs:
- `vision.md` — conceptual product vision
- `infra-outline.md` — infrastructure stack mental model
- `multiagent-deployment-frameworks.md` — ecosystem research

## 1. MVP scope (v1)

A hackathon-scoped, end-to-end demonstrable slice. Deliberately narrow.

**In scope:**
- One harness: **Hermes Agent** (Nous Research, from `ghcr.io/nousresearch/hermes-agent`)
- One deploy target: **Fly.io**
- Catalog-based spawn flow
- "RPG character creation" UX: name the agent, pick a model, paste API key, deploy
- Fleet view listing spawned agents with status
- Hand-written Hermes adapter

**Stubbed (not implemented, but scaffolded behind real interfaces):**
- Additional harnesses
- Additional deploy targets (AWS, SkyPilot, NixOS, local)
- Repo linking / zip upload / custom harnesses
- Programmatic adapter generation
- Model gateway
- Skills registry, tools, context, memory providers
- IAM, audit log, observability dashboard

## 2. Core concepts

### Harness
The code/image that wraps a model and exposes standard capabilities. **Internal
concept — not user-facing.** The user never sees the word "harness."

### Agent (user-facing)
The user-facing unit. Internally split into:

- **AgentTemplate** — reusable configuration shared by many instances. Bound
  to a specific `HarnessAdapter` at a pinned image digest. Holds default
  config schema.
- **AgentInstance** — one concrete deployed agent. Has its own name, owner,
  secrets, model provider, deployment state, and runtime identity.

This template/instance split is internal plumbing in v1 (there is only one
template, "Hermes"), but the data model encodes it from day one so v2 can
introduce multiple templates and user-defined templates without schema churn.

### HarnessAdapter
A thin shim layer that conforms an arbitrary harness to Corellia's contracts.
Implemented as a Docker image layered on top of the upstream harness image
(e.g., `corellia/hermes-adapter:sha-xyz` built `FROM nousresearch/hermes-agent:@sha256:...`).

### DeployTarget
Abstraction over infrastructure providers. Exactly one concrete implementation
in v1 (`FlyDeployTarget`). Every other target (`AWSDeployTarget`,
`SkyPilotDeployTarget`, `LocalDeployTarget`) is scaffolded as a real class that
raises `NotImplemented`, so that the interface is exercised by the MVP code
path.

## 3. The harness interface contract

The single most important architectural artifact in the project. Defines *how
Corellia talks to any harness, regardless of framework*. Structured as four
sub-contracts:

### 3.1 Runtime contract
What the running harness exposes over the network.
- HTTP server on a known port (convention: `8642` for Hermes; adapter may
  remap to a Corellia-standard port)
- `GET /health` — liveness probe
- `POST /chat` — OpenAI-compatible chat completion endpoint
- `POST /tools/invoke` — standardized tool-call entry point (post-v1)

### 3.2 Configuration contract
Env vars Corellia uses to configure *any* harness. The adapter translates
these to whatever the upstream harness natively expects.

- `CORELLIA_AGENT_ID` — stable identifier for this AgentInstance
- `CORELLIA_MODEL_PROVIDER` — `anthropic` / `openai` / `openrouter` / ...
- `CORELLIA_MODEL_API_KEY` — provider credential
- `CORELLIA_MODEL_NAME` — specific model (e.g., `claude-opus-4-7`)
- `CORELLIA_TOOL_MANIFEST_URL` — v1.5 Pillar B; URL the adapter polls for the
  per-instance toolset manifest (set as a Fly app secret at spawn time when
  `CORELLIA_API_URL` is configured on the control plane)
- `CORELLIA_INSTANCE_TOKEN` — v1.5 Pillar B; opaque per-instance bearer token
  the adapter sends with each manifest poll. SHA-256 hash stored in
  `agent_instance_manifest_tokens`; raw value lives only in the Fly app's
  secret store (blueprint §11.6)
- `CORELLIA_MEMORY_ENDPOINT` — post-v1

### 3.3 Packaging contract
- Distributable as a Docker image with a declared digest
- Known entrypoint (may be overridden by the adapter)
- Declared exposed ports
- Declared minimum resource footprint (CPU, RAM)

### 3.4 Metadata contract
A `corellia.yaml` manifest embedded in the adapter image (or repo) declaring:
- name, version
- required env vars and their descriptions
- default model
- supported tools (post-v1)
- resource requirements
- adapter version

For v1, this manifest is hand-written for Hermes. For v2+, it is generated by
the adapter-analysis pipeline and cached.

## 4. Adapter strategy

### v1 — hand-written
One adapter: `corellia/hermes-adapter`. Implemented as a small Dockerfile
that `FROM`s the upstream Hermes image, overrides the entrypoint with a
wrapper script that translates `CORELLIA_*` env vars to Hermes-native env
vars (e.g., `OPENROUTER_API_KEY`), then `exec`s Hermes.

### Post-v1 — programmatic generation
Pipeline:
1. Input: harness source (repo + commit SHA) or image reference.
2. Cache lookup on `{source_ref}`. If hit, return.
3. On miss: clone, parse structure with tree-sitter, feed structural map +
   README + Dockerfile to Opus, extract `corellia.yaml`, build adapter image.
4. **Validate**: spin up harness in sandbox, probe claimed endpoints, verify
   env var consumption.
5. Cache adapter image + manifest, keyed by source ref.
6. Regenerate on source-ref change.

### Data model support for both
The `HarnessAdapter` table includes a `source` column (`hand_written` |
`generated`) so hand-written and generated adapters coexist in the same UI
listing. Generated adapters carry a validation report; hand-written ones do
not. No schema change is needed when the analyzer ships.

## 5. Versioning

**Agent Templates pin by Docker image digest, never by mutable tag.**

- At template creation: resolve the chosen tag (e.g., `latest`, `v0.3.2`) to
  a concrete digest. Store the digest.
- Every AgentInstance spawned from that template uses the pinned digest. All
  instances of a template are bit-identical.
- Template upgrade is an **explicit, audited action**. Admin clicks upgrade,
  Corellia resolves the new tag to a new digest, template is updated.
- Existing AgentInstances continue running the old digest until rolled
  forward explicitly (per-instance or fleet-wide).

This is the foundational governance primitive: immutable artifacts + explicit
upgrades + audit trail.

## 6. Model routing

### v1 — direct
Each AgentInstance holds its own provider credentials (stored as Fly app
secrets). The agent calls the provider directly. Simple, no extra moving
parts.

### v2 — optional gateway
Introduce a LiteLLM-based model gateway. All agent model calls route through
it. Enables central rate limits, budgets, per-agent visibility, provider
swap, and cost auditing. Transition is transparent: the agent's
`CORELLIA_MODEL_ENDPOINT` env var flips from the provider's URL to the
gateway's URL. No agent-side code change.

**Do not build a gateway from scratch.** Host LiteLLM when the time comes.

## 7. Observability and memory integration

Integration patterns, ordered from least to most invasive:

- **Option C — Protocol hooks.** If the harness natively emits OpenTelemetry
  or webhooks, configure them at deploy time. Zero modification.
- **Option A — Sidecar container.** Co-located process that reads stdout,
  logs, or shared volumes and ships them to an external memory/observability
  provider. Zero modification.
- **Option B — Network proxy.** A wiretap on the agent's outbound network
  traffic. Captures every LLM and tool call. Zero modification, but requires
  running a proxy process.
- **Option D — Adapter wrapper (entrypoint shim).** The adapter itself runs
  setup logic at container boot. *Always used* for config translation in v1,
  and is the mechanism by which A/B/C get installed when needed. Minimal
  modification.
- **Option E — Fork the upstream harness.** Last resort. Avoid; upstream
  moves fast and forks rot quickly.

**Rule:** never fork an upstream harness. If a capability cannot be added via
A/B/C/D, it does not get added.

### v1 scope
Minimal: stdout capture via Fly's native log aggregation. Nothing more.

### Post-v1
Sidecar-based shipping to Elephantasm (or similar) for long-term agent
memory and full-conversation auditability.

## 8. Fly deployment topology

**One AgentInstance = one Fly app = one Fly machine.**

Rationale:
- Secrets are Fly-app-scoped; one-app-per-agent gives true per-agent secret
  isolation without custom multi-tenancy logic.
- Firecracker microVM isolation is per-machine, hypervisor-grade.
- Auto-stop + auto-start make idle agents effectively free.
- Clean destroy: `fly apps destroy <name>` removes everything.
- Simple mental model; easy to reason about and debug.

Cost/scale alternatives (one-app-many-machines, one-machine-many-agents) are
**not adopted in v1.** Revisited if and when Fly's app-count limits or
cost-per-idle-agent become a real constraint. At that point the correct move
is typically to add a second deploy target (bare-metal, NixOS, cheap VMs),
not to cram more agents into one Fly app.

### Machine config
- Firecracker microVM
- Auto-stop enabled, auto-start on request
- One container: the Corellia harness-adapter image, pinned by digest
- Secrets injected as environment variables at machine creation
- Volume: none in v1 (agent state is stateless or offloaded to memory provider)

## 9. Data model

Initial schema. All tables have `id` (UUID), `created_at`, `updated_at`
unless noted.

```
User
  email (unique)
  org_id
  role

Organization
  name

HarnessAdapter
  harness_name                  -- "hermes"
  upstream_image_digest         -- "sha256:..." of the upstream image
  adapter_image_ref             -- "corellia/hermes-adapter@sha256:..."
  manifest_yaml                 -- parsed corellia.yaml
  source                        -- enum: "hand_written" | "generated"
  generated_at                  -- nullable
  validated_at                  -- nullable
  validation_report             -- nullable JSON

AgentTemplate
  name
  description
  harness_adapter_id            -- FK → HarnessAdapter
  default_config                -- JSON, per-template defaults
  created_by_user_id

AgentInstance
  name                          -- admin-chosen
  agent_template_id             -- FK → AgentTemplate
  owner_user_id
  org_id
  deploy_target_id              -- FK → DeployTarget
  deploy_external_ref           -- e.g. "fly-app:alice-hermes-01"
  model_provider                -- "anthropic" | "openai" | "openrouter"
  model_name                    -- e.g. "claude-opus-4-7"
  status                        -- "pending" | "running" | "stopped" | "failed" | "destroyed"
  config_overrides              -- JSON, per-instance config
  last_started_at
  last_stopped_at

Secret
  agent_instance_id             -- FK → AgentInstance
  key_name                      -- e.g. "CORELLIA_MODEL_API_KEY"
  storage_ref                   -- reference to actual secret store entry
                                --   (not the raw value; secrets live in Fly's secret store)

DeployTarget
  name                          -- "fly" | "aws" | "skypilot" | "local"
  kind                          -- enum
  config                        -- JSON (region, defaults, etc.)
  enabled                       -- bool
```

## 10. User flow: "RPG character creation"

The golden path for v1.

1. Admin signs in.
2. Clicks "New Agent."
3. **Pick harness:** catalog shows "Hermes" (others grayed out / "Coming soon").
4. **Name:** free text.
5. **Pick model provider:** dropdown (Anthropic / OpenAI / OpenRouter).
6. **Paste API key:** stored as a per-instance secret.
7. **Pick model:** dropdown scoped to chosen provider.
8. **Deploy:** Corellia
   - Creates an `AgentInstance` row in state `pending`.
   - Calls `FlyDeployTarget.spawn(instance)`:
     - Creates Fly app (`corellia-agent-<uuid>`)
     - Sets app secrets from the instance's Secret rows
     - Creates one machine with the pinned adapter image
   - On Fly machine running + `/health` passing: flips state to `running`.
9. Admin is redirected to the fleet view. New agent appears with status
   and a link to its logs.

### "Spawn N agents" — the demo moment
The catalog screen supports a "Deploy N" action: same form, but instead of
one agent, the admin specifies count and a name prefix. Each spawn is an
independent Fly app creation, parallelized.

## 11. Architecture rules (codebase norms)

These are blocking rules. Breaking them is treated as a defect.

1. **No Fly-specific code outside `FlyDeployTarget`.** All Fly primitives
   (apps, machines, secrets) are accessed through that module. The rest of
   the codebase sees only the `DeployTarget` interface.
2. **AgentTemplate pins by digest, never by tag.** Tag-based references
   are a governance anti-pattern and are not accepted.
3. **Harness configuration flows through `CORELLIA_*` env var conventions.**
   Adapters translate; Corellia never reaches into a harness's native env var
   names from outside the adapter.
4. **Deferred features are stubbed as real interface implementations, not
   as fake UI buttons.** A `DeployTarget` that raises `NotImplemented` is
   acceptable; a UI button that does nothing is not.
5. **No forking of upstream harnesses.** Capabilities are added via adapter
   wrappers or sidecars, never by modifying upstream source.
6. **Deploy-target credentials never live in Corellia's database.** Raw
   credentials live in a secret store; DB rows reference them via opaque
   `storage_ref` (M4 decision 6, applied to deploy-target credentials the
   same way it applies to per-instance secrets). When v1.5 introduces
   user-supplied targets, the acquisition flow uses the provider's
   narrowest-capability mechanism — Fly OAuth → org-scoped macaroon, AWS
   STS → assumed-role with an external ID, etc. **Never accept PATs from
   users.** Paste-as-fallback is acceptable only when no narrower
   mechanism exists, with explicit capability scope labelled in the UI.
   The macaroon/role caveats define the capability contract; the
   adapter declares the minimum set it needs.

## 15. Strategic framing

**Deployment is a commodity; governance is the product.**

Every cloud and framework solves deployment one way or another. Management,
governance, access control, and fleet visibility across *heterogeneous*
agents is the real unsolved problem — and therefore where Corellia invests
engineering effort.

Two tests applied to any engineering decision:

1. **Is it reversible?** If yes (because a well-designed abstraction shields
   the rest of the code), pick the easiest option for today. If no, invest
   in making it reversible first.
2. **Does it differentiate the product?** If yes (harness contract, fleet
   view, IAM, audit), invest deeply. If no (VM provider, container runtime),
   minimum viable attention.

Most infra decisions fail test 2, which means they should get the minimum
viable attention.
