## 0.5.2 — Dev Tooling: Procfile Rename + Overmind Port Pinning (2026-04-26)

Polish-tier dev-tooling fix surfaced during a first-time-boot diagnosis on a fresh contributor environment. Two unrelated changes bundled because both block plain `overmind start` from working as documented in `stack.md` §7: (1) `Procfile.dev` renamed to `Procfile` so overmind's default discovery (`./Procfile`) works without `-f` or a pre-set `OVERMIND_PROCFILE`, (2) the Procfile itself rewritten to pin FE port 3000 and BE port 8080 explicitly, defending against overmind's auto-`PORT` injection that silently re-binds first-in-Procfile processes per the foreman/Heroku convention. **Supersedes 0.2.0**'s introduction of `Procfile.dev` as a Heroku-style namespace separator — Vercel ignores Procfile entirely, and Fly.io reads `fly.toml`'s `[processes]` block, so the suffix was guarding against a deploy collision that can't happen. No runtime change; no code outside repo-root tooling files; the diff is one `git mv` + one Procfile rewrite + four live-doc reflows. Patch version (not minor) per the 0.5.1 precedent for sub-milestone polish; thematically separate from 0.5.1's M3.5 product work.

### Index

- **`Procfile.dev` → `Procfile` (via `git mv`).** *Where:* repo root. The `.dev` suffix was a defensive holdover from Heroku's deploy convention (a plain `Procfile` is read by Heroku's build system as the production process declaration; separating dev orchestration prevented `next dev` and `air` from accidentally running in production). Corellia's deploy path is Vercel + Fly.io — neither reads Procfile — so the suffix was guarding against a collision that can't happen. Cost of keeping it: every `overmind start` needed `-f Procfile.dev` or a pre-set `OVERMIND_PROCFILE`, and `stack.md` §7 + `CLAUDE.md` §Common commands both already documented the bare command. **The rename resolves the doc-vs-file inconsistency in the direction the docs already assumed** — `git mv` preserves history so `git log --follow Procfile` traces back through the `.dev` era.
- **Procfile rewritten to pin both ports explicitly.** *Where:* `/Procfile` — was `web: pnpm -C frontend dev` + `api: cd backend && air`; now `web: pnpm -C frontend dev --port 3000` + `api: cd backend && PORT=8080 air`. *Why:* overmind injects `PORT=<base+stride>` into each process per foreman/Heroku convention (default base 5000, +100 per process), which without explicit overrides would shift FE→5000 and BE→5100. On Linux this would have produced "FE loads, can't reach BE, network errors throughout the catalog page" — the FE bakes `NEXT_PUBLIC_API_URL=http://localhost:8080` into its bundle at build time. On macOS Sequoia the failure is loud (`EADDRINUSE :::5000`) because the AirPlay Receiver squats on port 5000 by default, which is operationally lucky — surfaces at boot rather than at debug time. **Pinning the ports explicitly is idiomatic, not a workaround**: Next.js's `--port` flag wins over `process.env.PORT`, and `PORT=8080 air` exports the value into `air`'s child env so `internal/config.Load()` reads `8080` regardless of what overmind injects. The committed Procfile now matches what `FRONTEND_ORIGIN`, `NEXT_PUBLIC_API_URL`, and `internal/config.Config.PORT`'s default already assumed.
- **Live docs reflowed across four files.** `README.md` (×2: stack table + repo-layout tree), `CLAUDE.md` (Common commands code block at line 113), `docs/stack.md` (×3: stack table §1, layout tree §2, hour-zero scaffolding step §12), `docs/blueprints/toolchain-overview.md` (overmind tool description). Historical references in `docs/changelog.md` (the v0.2.0 entry's mention on line 879, plus the architecture-diagram update note around line 829) and `docs/archive/*` are deliberately untouched — the changelog convention "when a decision contradicts an earlier one, note the supersession in the new entry rather than editing the old one" applies. Future readers grepping `Procfile.dev` in those archived contexts will see it as the historical name and find the supersession via this entry.

### Behavior change (known)

- **`overmind start` from repo root now works without `-f Procfile.dev`.** The implicit assumption in `stack.md` §7 (which documented the bare command pre-fix) is now literal.
- **First-process `PORT` injected by overmind no longer changes FE bind port.** `pnpm -C frontend dev --port 3000` overrides whatever overmind sets — Next.js's CLI flag has higher precedence than `process.env.PORT`. Same shape on the BE: `PORT=8080 air` exports the value into `air`'s child env so `internal/config.Load()` reads it before overmind's injection takes effect.
- **No runtime change in production.** Vercel and Fly.io don't read Procfile; this entry touches dev orchestration only.
- **No new env vars, tools, or dependencies.** A pure repo-root tooling fix.

### Resolves

- **First-time-boot friction on macOS.** Any contributor with AirPlay Receiver enabled (default on Sequoia) would have hit `EADDRINUSE :::5000` on their first `overmind start`. Pinning the FE port removes a class-of-friction, not a single instance.
- **Doc-vs-file inconsistency in the local-dev story.** Pre-0.5.2 the docs said `overmind start`; the file required `-f Procfile.dev`. Post-0.5.2 they match — `overmind start` is literal, not aspirational.

### Supersedes

- **0.2.0's introduction of `Procfile.dev`** (changelog line 879 — "Monorepo workspace plumbing: root `pnpm-workspace.yaml` + `package.json` + `Procfile.dev`"). The Heroku-style namespace separator was defensive against a deploy convention Corellia never adopted. Plain `Procfile` is the canonical name from 0.5.2 onward.
- **The implicit pre-0.5.2 contract that `overmind start` finds `Procfile.dev`.** It didn't — operators had to pass `-f Procfile.dev` or pre-set `OVERMIND_PROCFILE`. `stack.md` §7's bare `overmind start` was the desired-not-actual command pre-0.5.2; from 0.5.2 it's literal.
- **The pre-0.5.2 Procfile shape** (`web: pnpm -C frontend dev` + `api: cd backend && air`). The new shape with explicit `--port 3000` and `PORT=8080` is the canonical form; any future Procfile edit should preserve the explicit-port-pinning pattern.

## 0.5.1 — M3.5: Deploy Target Resolver Indirection (Phases 1–4) (2026-04-26)

Structural follow-up to M3's `internal/deploy/` package — **no behavior change**. Where 0.5.0 made the `DeployTarget` interface concrete (one Fly-backed `FlyDeployTarget`, two `NotImplemented` stubs, a kind-keyed `map[string]DeployTarget` exposed on `httpsrv.Deps`), 0.5.1 inserts one indirection layer between handlers and that map: a new `deploy.Resolver` interface with a today-implementation (`StaticResolver`) that wraps the same env-var-bootstrapped registry. Behavior is byte-identical at boot — same flaps client construction, same three targets registered, same `kinds=aws,fly,local` boot log — but the field on `httpsrv.Deps` is now narrower (`deploy.Resolver` instead of `map[string]DeployTarget`), the constructor accepts a struct instead of two positional strings (`FlyCredentials`), and the env vars that bootstrap it carry a `// TODO(v1.5):` retirement breadcrumb. Patch version (not minor) per the 0.3.0 → 0.3.1 M1-hardening precedent: this is structural pre-payment toward v1.5's user-configurable deploy targets, not a product feature. Plan: `docs/executing/deploy-target-resolver.md` (locked ahead of execution). Four per-phase completion docs under `docs/completions/deploy-target-resolver-phase-{1..4}.md`.

The whole milestone is +120 / -12 across 5 files (one new file, four edits) and lands in monotonically-decreasing diff sizes — Phase 1 +20/-5, Phase 2 +88/-1, Phase 3 +2/-2, Phase 4 +9/-3. Each phase's diff size measures how much of the architectural pre-payment was already in place when it landed; Phase 3's two-line diff is the load-bearing evidence that Phases 1 and 2 did their jobs. The eventual v1.5 swap (`StaticResolver` → `DBResolver`) is now a one-line type change in `cmd/api/main.go` plus deletion of the two annotated `Config` fields — zero handler updates, because no handler reaches into the underlying map.

### Index

- **`deploy.FlyCredentials struct{APIToken, OrgSlug string}` replaces the positional `(token, orgSlug)` constructor pair** (Phase 1). `NewFlyDeployTarget(ctx, FlyCredentials{...})` is the new shape; both call sites (`cmd/api/main.go:50`, `cmd/smoke-deploy/main.go:49`) updated to named-field struct literals. **The struct is shape, not capability** — exactly two fields, byte-identical to the previous arguments — but named-field literal syntax makes future additive growth (DefaultRegion, scoped tokens, MaxConcurrentSpawns) cost zero at existing call sites. No deprecation period, no shim function: a single-commit constructor break across three files. Plan-time pre-work assumed two grep hits but execution found three (`cmd/smoke-deploy` was added in M3 Phase 7, after the resolver plan was drafted); plan §risk-register entry 4 had explicitly anticipated this and the third caller was absorbed inline rather than spinning out a Phase 1.5.
- **New `deploy.Resolver` interface and `StaticResolver` implementation** (Phase 2). `Resolver` is single-method: `For(ctx context.Context, kind string) (DeployTarget, error)`. The ctx argument is unused by `StaticResolver` but required by v1.5's `DBResolver` for the row fetch + decryption call. **The interface deliberately omits `Kinds()` / `List()`** — adding a list method would be premature interface widening for a single boot-time observability call; the `slog.Info("deploy targets initialised", "kinds", "aws,fly,local", ...)` line keeps reading from the underlying map by design (Phase 3 carve-out). New file `internal/deploy/resolver.go` (63 LOC) holds the interface, the `StaticResolver` struct, the constructor, the `For` method, and a compile-time assertion (`var _ Resolver = (*StaticResolver)(nil)`) — the same pattern `target_test.go` already uses for `DeployTarget` conformance.
- **New `deploy.ErrTargetNotConfigured` sentinel, distinct from `ErrNotImplemented`** (Phase 2). Two sentinels, two semantics: `ErrNotImplemented` (in `target.go`, M3) means "this target type exists as a stub but its methods aren't built yet" — the blueprint §11.4 stub-vs-real distinction for `LocalDeployTarget` / `AWSDeployTarget`. `ErrTargetNotConfigured` (in `resolver.go`, M3.5) means "the resolver has no entry for this kind" — a different operator failure with a different M4 spawn-handler response. **Conflating them would give the spawn handler one ambiguous error to log and react to.** Test cases pin the contract via `errors.Is(err, ErrTargetNotConfigured)` rather than equality, so a future wrapping via `fmt.Errorf("...%w", ErrTargetNotConfigured)` won't break the test.
- **`httpsrv.Deps.DeployTargets` narrowed from `map[string]deploy.DeployTarget` to `deploy.Resolver`** (Phase 3). Single-line diff in `internal/httpsrv/server.go`'s `Deps` struct — field name and ordering preserved. **No handler updates needed because no handler reads the field yet** (M3 Phase 6's "wire the field, don't consume it" discipline holds); M4's spawn handler will be the first reader and will consume `resolver.For(ctx, kind)`, never `deps.DeployTargets[kind]`. The pre-work grep (`grep -rn 'DeployTargets\[' backend/internal/httpsrv/`) returned zero hits at the start of every phase 1–4, confirming the discipline survived end to end.
- **`cmd/api/main.go` constructs `*StaticResolver` at boot, between the existing map construction and `httpsrv.New(...)`** (Phase 3). One new line — `deployResolver := deploy.NewStaticResolver(deployTargets)` — sits *after* the `slog.Info("deploy targets initialised", ...)` so the boot sequence reads top-to-bottom as "build map → log map's contents → wrap map in resolver → pass resolver to handlers." The `keysOf(deployTargets)` slog still operates on the map, not the resolver, preserving Phase 8's `sort.Strings`-driven deterministic `kinds=aws,fly,local` runbook output. **The map outlives its registry role only as a transient scaffold for the slog line** — a deliberate non-orthogonality that costs one extra local-variable lifetime and avoids premature interface widening.
- **`Config.FlyAPIToken` and `FlyOrgSlug` carry a six-line block comment plus a `// TODO(v1.5):` retirement breadcrumb** (Phase 4). Comment-only edit at `internal/config/config.go:31-37` — no field rename, no env-tag change, no consumer update. **The version-tagged TODO is structurally different from a generic TODO**: scoped to a specific milestone (the v1.5 `DBResolver` swap), self-superseding (the deletion of the TODO and the deletion of the fields happen in the same commit), greppable as the canonical v1.5 punch list (`grep -rn 'TODO(v1.5)' backend/` returns this single line, and will return the v1.5 to-do list as more phases use the convention). Establishes the version-tagged-TODO convention in the codebase; future provisional state should use the same form.
- **Compile-time assertion as defense-in-depth across three sites.** The `var _ Resolver = (*StaticResolver)(nil)` line in `resolver.go` (Phase 2) is what makes the `httpsrv.Deps.DeployTargets deploy.Resolver` field at server.go:21 (Phase 3) accept a `*StaticResolver` value with no explicit conversion at the `cmd/api/main.go` callsite. **The three sites form a contract pin** that catches signature drift on `Resolver.For` at the most useful diagnostic location: changes to the interface fail at the assertion; changes to the implementation fail at the assertion; changes to the field type fail at the callsite. Same pattern M3's `target_test.go` established for `DeployTarget` (Phase 5 of M3); this milestone applies it to one additional interface.
- **Validation matrix.** `cd backend && go vet ./... && go build ./... && go test ./...` all clean — `internal/deploy` flipped from 24 sub-tests (M3 Phase 8) to 26 sub-tests (Phase 2's two new resolver cases); `internal/agents` and `internal/users` cached at their respective M2 / 0.2.5 baselines; both binaries (`cmd/api` and `cmd/smoke-deploy`) build end-to-end. `grep -rn 'DeployTargets\[' backend/` returns zero hits — the field-narrowing didn't surface any latent direct-map consumers. `grep -rn 'NewFlyDeployTarget(ctx, cfg\.' backend/` returns zero hits — the old positional constructor form is fully retired. Boot-time runtime smoke (`cd backend && air`) deferred to runbook per the M3 precedent that runtime checks are deploy-confidence gates, not merge gates.
- **Plan-as-built drift on Go API code: zero this milestone.** All five plan-prescribed Go signature changes (struct definition, interface declaration, two method bodies, two callsite swaps) landed exactly as drafted — no SDK-version drift, no fly-go API surprises, no signature reshape in execution. **The plan's structural decisions and call-site prescriptions both survived intact** because the Go surface this plan touches is internal — no third-party SDK godoc to drift against. Contrast with M3 where 3/7 phases needed signature touch-up against `fly-go` v0.5.0; the resolver plan's all-internal scope was structurally easier to land on the prescribed shape.

### Phase 1 — `FlyCredentials` + constructor swap (`docs/completions/deploy-target-resolver-phase-1.md`)

**Smallest meaningful structural change in the codebase to date** — three file edits, +20 / -5 LOC, one new exported type, one constructor signature change, zero new files, zero new tests. The smallness is the point: the structural pre-payment is supposed to be cheap *now* so the v1.5 swap is cheap *later*. At 20 LOC, the option's premium is barely measurable — and the option (constructor signature never changes again, regardless of how many credential fields v1.5 adds) is unconditionally valuable.

**Plan §risk-register entry 4 paid off the way good risk registers do** — not by preventing the third-caller surprise but by pre-deciding what to do when it landed. Pre-work step 2 expected exactly two `NewFlyDeployTarget` grep hits (definition + one caller); the actual count was three because `cmd/smoke-deploy/main.go` was added in M3 Phase 7, *after* the resolver plan was drafted. The plan author's grep was correct at draft time but stale at execution time. The risk-register entry's framing ("If a third appears between drafting and execution, add a Phase 1.5 to update it") was the escape hatch; since both callers needed the same one-line edit, no escape was needed and the third caller absorbed inline.

**Both call sites use named-field struct literal syntax** (`deploy.FlyCredentials{APIToken: ..., OrgSlug: ...}`), not positional. Named-field syntax is forward-compatible — adding a field to `FlyCredentials` doesn't break callers that don't supply it (Go zero-values the omitted field). Positional syntax would break every caller the moment a third field lands. **The named-field convention is what makes the struct's "additive growth" promise actually deliverable**; positional callers would defeat it on the first extension.

### Phase 2 — `Resolver` interface + `StaticResolver` impl + 2 test cases (`docs/completions/deploy-target-resolver-phase-2.md`)

**One new file (`internal/deploy/resolver.go`, +63 LOC) and one test extension (`target_test.go`, +25 / -1).** The file holds the `ErrTargetNotConfigured` sentinel, the `Resolver` interface, the `StaticResolver` struct + constructor + `For` method, and a compile-time assertion at the bottom. Doc comments on `ErrTargetNotConfigured`, `Resolver`, and `StaticResolver` cite blueprint §11.4 and the resolver plan's decision 2 by name — the comment-as-breadcrumb pattern Phase 1's completion doc established.

**`*StaticResolver` returns `*StaticResolver`, not `Resolver`, from its constructor.** Go-idiomatic ("accept interfaces, return structs") and matches the M3 pattern (`NewLocalDeployTarget()` returns `*LocalDeployTarget`, etc.). Test code can reach into struct internals if needed without a type assertion; the compile-time assertion at the bottom of `resolver.go` enforces interface conformance regardless of the return type. **Convention over indirection** — the assertion is the load-bearing check, not the return type.

**Two test cases use `errors.Is`, not equality, against `ErrTargetNotConfigured`.** Writing `if err != ErrTargetNotConfigured` would pass today (the sentinel is returned bare), but `errors.Is` survives a future refactor where `For` wraps the sentinel via `fmt.Errorf("...%w", ErrTargetNotConfigured)` to add a kind string to the message. **The test pins the contract (sentinel is reachable via `errors.Is`), not the current implementation (sentinel is returned bare).** Same pattern the M3 domain services use for redacted-error returns.

**`LocalDeployTarget` reused as the registered-target fixture** rather than introducing a new test-only fake. `LocalDeployTarget` already implements `DeployTarget` in the same package; its `ErrNotImplemented` returns are irrelevant because `For` doesn't call any of its methods. **Reusing the stub keeps the test file dependency-free of new types** — no `fakeTarget struct{}` declaration, no scope creep into "what does a minimum DeployTarget need to look like."

### Phase 3 — `httpsrv.Deps` field narrowing + `cmd/api` wiring (`docs/completions/deploy-target-resolver-phase-3.md`)

**Smallest meaningful diff in the resolver plan** — +2 / -2 across two source files. One identifier change in `Deps` (`map[string]deploy.DeployTarget` → `deploy.Resolver`) and one new line in `main.go` (`deployResolver := deploy.NewStaticResolver(deployTargets)`) plus the `httpsrv.Deps{...}` literal swap. **The diff size is the load-bearing evidence that Phases 1 and 2 did their jobs**: Phase 1's `FlyCredentials` decoupled the constructor from the credentials' shape; Phase 2's `Resolver` interface + `StaticResolver` impl gave Phase 3 a type to swap to. A 2-line consumer-adoption phase would have been structurally impossible without the prior 88-line abstraction phase.

**The `slog.Info` line still reads from the map, not the resolver** (plan §Phase 3 step 3). The runbook contract from M3 Phase 8 (`kinds=aws,fly,local` deterministic order via `sort.Strings(keysOf(deployTargets))`) keeps working unchanged because `keysOf` operates on the map; the resolver doesn't expose a `List`/`Kinds` method by design. **The map outlives its registry role only so the boot log can introspect it** — a deliberate carve-out, not a smell. Strict adherence to "the map is dead post-Phase-3" would force either premature interface widening or losing the deterministic boot log; keeping the map as a transient scaffold loses neither architectural property.

**`grep -rn 'DeployTargets\[' backend/internal/httpsrv/` returns zero hits — same as Phase 2's pre-work confirmed.** M3's "wire the field, don't consume it" discipline held through Phase 3. **The window of cheap reversibility closes when M4 lands** — once the spawn handler reads `deps.DeployTargets.For(ctx, kind)`, reverting to a raw-map shape would require updating that handler too. M4's plan is the latest opportunity to revisit the design choice.

### Phase 4 — `config.go` bootstrap-state annotation (`docs/completions/deploy-target-resolver-phase-4.md`)

**Comment-only edit, +9 / -3 LOC, no field rename, no env-tag change, no consumer update.** Six-line block comment plus a one-line `// TODO(v1.5):` immediately above `FlyAPIToken`/`FlyOrgSlug`, plus a separating blank line that splits the env block into two visual groups: "current state" (Port, DatabaseURL, SupabaseURL, FrontendOrigin) and "bootstrap state slated for v1.5 retirement" (FlyAPIToken, FlyOrgSlug). The blank-line split is informationally redundant with the comment but pays back at every re-read.

**The annotation lives at the field declaration site, not the call sites.** Two sites consume `cfg.FlyAPIToken`: `cmd/api/main.go:51-52` and `cmd/smoke-deploy/main.go`. Putting the breadcrumb at the field declaration means a future contributor reading "what does `Config.FlyAPIToken` mean?" lands on the rationale immediately, regardless of which call site they navigated from. **The field declaration is the single source of truth for the field's lifecycle**; call sites describe consumption, not provenance.

**`// TODO(v1.5):` is a greppable retirement marker, not a generic "fix this later."** Three load-bearing properties: (a) scoped to a specific milestone, (b) distinguishable from generic TODOs in any future grep audit, (c) self-superseding (the deletion of the TODO and the deletion of the fields happen in the same commit). **A bare `TODO` would have none of these properties.** Establishes the version-tagged-TODO convention in the codebase; future provisional state should follow the same form.

**No mirror annotation in `.env.example`.** A parallel "this is bootstrap state slated for v1.5 retirement" annotation could have been added to the operator-facing template, but: operators need to know what to set (not whether the field is provisional); when the v1.5 deletion lands, both `.env.example` lines disappear in the same commit anyway; one canonical breadcrumb in `config.go` is the better single source of truth than scattered TODOs across two files. Decision parallels Phase 3's "no `// TODO(v1.5):` at the resolver-wrapping line" reasoning — one canonical breadcrumb beats many.

**Phase 4 is the first M3 phase whose `goose up` makes a durable state change to the running database.** New migration `backend/migrations/20260426120000_adapter_image_ref_backfill.sql` (56 LOC, single transaction): UPDATE writes Phase 2's captured digest, ALTER tightens to `NOT NULL` with `CHECK (adapter_image_ref LIKE '%@sha256:%')` named `adapter_image_ref_digest_pinned`. Migration applied cleanly in 53ms; goose's per-file atomicity means no partial-state window between UPDATE and ALTER.

**Both rejection paths verified empirically, not just by inspection.** A tag-form UPDATE (`'ghcr.io/...:latest'`) was rejected with `ERROR: ... violates check constraint "adapter_image_ref_digest_pinned"` — the digest-pinning CHECK is live. A NULL UPDATE was rejected with `ERROR: null value in column "adapter_image_ref" ... violates not-null constraint` — both constraints are independently enforced. Down/up cycle round-trips cleanly (`goose down` 105ms → empty column → `goose up` 87ms → digest re-populated).

**`AdapterImageRef *string` → `string` is the load-bearing type-flip.** sqlc's regen flipped the field to bare `string` post-migration; Phase 5's `FlyDeployTarget.spawn(...)` sees the non-pointer type from the start — no nil-check at the call site, no `*ref` deref ceremony. **This is type-driven design at the database layer**: tighten the schema to the strongest invariant the application needs, and the generated types eliminate entire classes of error-handling branches at the call site for free.

**CHECK pattern uses `LIKE '%@sha256:%'`, deliberately permissive on registry host.** Pinning `'ghcr.io/%@sha256:%'` would lock in the operational choice of where the image lives; what matters for §11.2 is the *governance property* (digest-pinned), not the location. Survives a future move to Fly's own registry, self-hosted Harbor, or an internal Corellia registry with no schema migration. Same reasoning M2 used for `upstream_image_digest LIKE 'sha256:%'` — both columns now have permissive-but-strict CHECKs.

**Audit-comment block expanded over the plan's sketch.** The implemented comment adds `built_from` (the upstream Hermes digest from Phase 1/2) and a paragraph about why the manifest-list digest is non-deterministic across rebuilds. Future readers can answer "where did this digest come from, and why does the next rebuild produce a different one?" entirely from the SQL file. **Audit-comment-as-archaeology pattern**: any migration that imports an externally-captured fact (a digest, a Cosign certificate identity, a third-party webhook secret) should carry the capture provenance in a SQL comment. The migration corpus doubles as a forensic trail.

**Phase 4 is the smallest M3 phase by LOC and the largest by governance impact.** 56 LOC of SQL + 1-line generated-code type-flip closes blueprint §11.2's enforcement gap on the *adapter* digest column. Pre-Phase-4, §11.2 was fully encoded for the *upstream* digest and partially encoded for the *adapter* digest (Go-level convention only); post-Phase-4, both have CHECK constraints an operator with `psql` access cannot bypass without explicit constraint-violation.

### Phase 5 — `internal/deploy/` package + `adapters.UpdateImageRef` (`docs/completions/hermes-adapter-and-fly-wiring-phase-5.md`)

**Phase 5 is the first M3 phase whose deliverable is Go code that calls Fly's API at runtime.** New package `backend/internal/deploy/` (target.go 55 LOC + fly.go 180 LOC + stubs.go 34 LOC + target_test.go 11 LOC). Per blueprint §11.1, this is the *only* place in the codebase that imports `fly-go` or talks to Fly; every other package sees only the `DeployTarget` interface. The `DeployTarget` interface has five methods (`Kind` / `Spawn` / `Stop` / `Destroy` / `Health`); `Logs` (streaming) deferred to v1.5.

**The plan's `fly-go` API surface had drifted from v0.5.0 reality in three places, all caught at write-time by reading the actual godoc.** (1) `fly.NewClient(fly.ClientOptions{Tokens: fly.TokensFromString(...)})` doesn't exist — actual constructors are `fly.NewClient(accessToken, name, version, logger Logger) *Client` and `fly.NewClientFromOptions(opts ClientOptions) *Client`; tokens live in `github.com/superfly/fly-go/tokens.Tokens` via `tokens.Parse(string) *Tokens`. (2) `(*fly.Client).SetSecrets(ctx, app, env)` doesn't exist — secrets are a per-key flaps round-trip via `(*flaps.Client).SetAppSecret`. (3) `flaps.NewClientOpts` has no `AppName` field — flaps clients are global; per-app routing is a method argument.

**flaps-only beats fly-graphql + flaps for v1's needs.** Plan's sketch held both a `*fly.Client` (GraphQL) for `CreateApp` / `DeleteApp` / `SetSecrets` and a `*flaps.Client` for `Launch` / `List` / `Stop`. Phase 5 collapsed this to flaps-only because: `flaps.CreateAppRequest{Org: <slug>}` accepts the org slug directly (no GraphQL ID resolution); `flaps.DeleteApp(ctx, name)` exists as the symmetric counterpart of `flaps.CreateApp`; one transport means one credential plumbing path, one HTTP client setup, one error surface to map. **§11.1 compliance is unchanged**: `internal/deploy/` is still the only package importing either name.

**`appNameFor` uses UUIDv5(`NameSpaceURL`, "corellia/<name>") for non-UUID inputs.** Deterministic for a given input string, no central registry needed; eight-character app-name suffix gives 16⁸ ≈ 4.3B distinct prefixes (collision-resistant for the v1/v2 sizing target). For UUID inputs (the production hot path: an `AgentInstance.id` from the DB), the function recognizes the format via `uuid.Parse` and uses the parsed UUID's first 8 hex chars directly. **`validateImageRef` is a defence-in-depth check**, not a primary validation — the Postgres CHECK from Phase 4 is the authoritative enforcement; the Go function catches programmer error before the API call is made.

**`Spawn` does *not* wait for the machine to reach `started`.** Returns immediately after `flaps.Launch` — caller persists `SpawnResult{ExternalRef, MachineID}` and is expected to poll `Health` if it cares. The HTTP request shouldn't block on Fly's scheduling; the polling loop belongs in the same place as the AgentInstance status update (M4), not split across two abstraction layers. **`Stop` lists then iterates** rather than firing a single app-level Stop call — flaps has no app-scoped stop primitive, every flaps stop is per-machine; v1's "one AgentInstance = one Fly app = one Fly machine" topology means single-iteration in practice but the shape generalises to v2's "multiple machines per app." **`Destroy` is `flaps.DeleteApp`, not `Stop` then `Delete`** — `flaps.DeleteApp` cascades; calling `Stop` first would be courtesy gentleness Hermes 0.x doesn't need (no in-flight transactions, no socket to drain).

**`adapters.UpdateImageRef` extends M2's `adapters.Service`.** New SQL query `UpdateHarnessAdapterImageRef :one` co-located with M2's `GetHarnessAdapterByID` in `harness_adapters.sql`; new domain method `UpdateImageRef(ctx, id, ref) (db.HarnessAdapter, error)` follows the established `pgx.ErrNoRows → ErrNotFound` redacted-error pattern. **Three services, one architectural pattern** — adding a fourth in M4 (`agents.SpawnInstance`) is paint-by-numbers.

**`target_test.go` is the smallest test file in the codebase doing the most work per LOC.** Eleven lines, three `var _ DeployTarget = (*X)(nil)` declarations. **The interface assertion is the cheapest enforcement mechanism for §11.1 available in Go** — any future PR that renames or removes `Spawn`/`Stop`/`Destroy`/`Health`/`Kind` on any concrete type produces a *directed* build failure on the exact line that pins the contract. (Phase 8 widens this file with 24 functional sub-tests; the compile-time assertions stay.)

### Phase 6 — `cmd/api/main.go` wiring + `httpsrv.Deps.DeployTargets` (`docs/completions/hermes-adapter-and-fly-wiring-phase-6.md`)

**Phase 6 is the first M3 phase whose deliverable is the `cmd/api` binary actually constructing a `*deploy.FlyDeployTarget` at boot.** Two-file diff: `internal/httpsrv/server.go` (+2 LOC: one new `DeployTargets` field on `Deps` between `AgentsHandler` and `AllowedOrigin`, one new import) + `cmd/api/main.go` (+25 / -1 LOC: two new imports, five constructor lines, one slog line, one `Deps`-field assignment, one 6-LOC `keysOf` helper).

**The `slog.Info("deploy targets initialised", "kinds", "aws,fly,local", "fly_org", <slug>)` line is the operator's single point of confirmation that Phase 6 fired.** Three lines now appear in order at boot: `jwks initialised` → `deploy targets initialised` → `listening addr=:8080`. An operator running `cd backend && air` confirms the Fly token+slug are non-empty (config didn't panic), `deploy.NewFlyDeployTarget` succeeded (flaps client constructed), and all three targets registered in the map. **Three log lines = three subsystems confirmed, no ambiguity.**

**`map[string]deploy.DeployTarget` keyed by `Kind()` is structurally a polymorphic dispatch table.** The map keys come from each target's own `Kind()` method — a self-describing registry, not a hand-maintained string table. The boot log emits all three kinds, advertising the full surface to anyone reading the logs. **The map is the polymorphic dispatch table that blueprint §11.4 implicitly mandates.**

**`_ = adaptersSvc` is the M3-tier compromise between two real codebase rules.** CLAUDE.md says "don't add features beyond what the task requires" (would forbid wiring `adapters.NewService` with no caller). The M3 plan says "pre-wire now to avoid M4-time `main.go` churn." The blank-identifier line satisfies both: service constructed at boot-time (M4 will replace `_ =` with a real assignment in one keystroke); Go's unused-variable rule is satisfied. **Phase 8 deletes this** in favor of CLAUDE.md's stricter "delete unused code completely" stance — see Phase 8 below.

**Constructor pattern: explicit `localTarget := deploy.NewLocalDeployTarget()` once each, not the plan's double-call shape.** Plan had `deploy.NewLocalDeployTarget().Kind(): deploy.NewLocalDeployTarget()` (calling each constructor twice per stub); as-built constructs each stub once and uses the variable on both sides of the map literal: `localTarget.Kind(): localTarget`. Generalises better when a stub gains state (config, sentinel error).

**Error handling on `NewFlyDeployTarget` is `slog.Error + os.Exit(1)`, not the plan's implicit success assumption.** The plan's sketch had `flyTarget := deploy.NewFlyDeployTarget(...)` (single return value); as-built honors Phase 5's `(*FlyDeployTarget, error)` signature with the standard boot-error pattern. Cross-phase plumbing that only surfaces at integration time — neither phase plan had it because each was reasoning about its own boundary in isolation.

**`grep -rn DeployTargets backend/internal/httpsrv/` returns exactly one match** (the field declaration); zero references in any handler. **Phase 6 left no half-finished plumbing** — the field's job in M3 is to be reachable from `New(d Deps)`, and `Deps`-parameter destructuring is the only edge it needs to cross. M4 is the first reader.

### Phase 7 — `cmd/smoke-deploy/` Go-level harness contract validation (`docs/completions/hermes-adapter-and-fly-wiring-phase-7.md`)

**Phase 7 is the first M3 phase whose deliverable is an executable that calls Phase 5's package against the real Fly API end-to-end.** New `backend/cmd/smoke-deploy/main.go` (108 LOC) constructs a `*deploy.FlyDeployTarget` exactly the way `cmd/api/main.go` does at boot, then walks it through Spawn → Health-poll → Destroy with the same Phase-4-pinned digest the shell smoke defaults to. **The two smokes (shell + Go) exercise *complementary* API paths** — `flyctl` on one side, `fly-go` + `flaps` on the other — which means a green from both proves both paths work and a green from one with a fail from the other isolates whether the bug lives in the codebase's wrapper code (`fly.go`) or in the underlying API path.

**The smoke driver lands as a permanent committable artefact, not a gitignored scratch file.** Two reasons: (a) the alternative — gitignoring it — means every operator who needs to smoke-test re-derives the file from the plan, and the plan's Go snippet keeps drifting; (b) `cmd/smoke-deploy/main.go` is a *contract test* between the codebase and the live Fly API in the same way `adapters/hermes/smoke.sh` is a contract test against the published image. **Two smoke artefacts, one repo, two complementary contract tests.**

**Three guardrails the as-built smoke adds beyond the plan's literal sketch.** (1) **Pre-flight refusal to spawn without `CORELLIA_SMOKE_API_KEY`** — without the check, `flaps.Launch` would succeed, the harness would boot, then crash on first model-call attempt because `CORELLIA_MODEL_API_KEY` is empty; "refused to spawn" is easier to debug than "machine briefly Started then Failed." (2) **Image ref via `CORELLIA_HERMES_ADAPTER` env var with Phase-4 digest as default** — `defaultImageRef` const matches `adapters/hermes/smoke.sh`'s `IMAGE` default exactly so a digest bump is a coordinated edit across both files. (3) **Early-exit on `HealthFailed`** — short-circuits on terminal failure (no `failed → started` transition without external action); deferred `Destroy` still fires.

**`defer target.Destroy(ctx, res.ExternalRef)` is the same shape `smoke.sh` uses with `trap '... fly apps destroy ...' EXIT`** — but Go's `defer` does *not* fire on `os.Exit` (which the smoke calls explicitly on three failure paths). As-built routes `os.Exit(1)` only after the deferred Destroy has had a chance to run by structuring success/fail signalling around `return` (success arm) and a final `os.Exit(1)` *after* the loop's `time.Sleep` exhausts (timeout arm). **The one path that *would* skip Destroy is `panic`** — but no panic site exists in the smoke; every error gets a structured stderr line with the deferred cleanup already queued.

**The smoke binary imports `internal/config` and `internal/deploy` and nothing else from the codebase.** No DB, no JWKS, no Connect, no domain packages. **The cheapest possible binary that proves Phase 5's package works end-to-end against real Fly.** 11.9MB binary size vs `cmd/api`'s 27.7MB reflects the smaller transitive dependency closure.

**`config.Load()` is invoked even though only two fields are read.** `cfg.FlyAPIToken` and `cfg.FlyOrgSlug` are the only fields the smoke uses; `DatabaseURL`, `SupabaseURL`, `FrontendOrigin` are still required env vars that `config.Load()` will panic on if missing. **The fix is not to relax `config.Load`'s validation** (it's correct for `cmd/api`'s purposes); the operator runbook documents that `backend/.env` must be fully populated. Future hardening: a separate `FlyConfig{}` struct for smoke-only required fields is a 20-LOC follow-up if friction surfaces.

**The plan-vs-as-built drift on Go API code has now happened in three out of seven M3 phases** (Phase 4 type-flip, Phase 5 fly-go SDK, Phase 7 smoke driver). **The plan got the architecture right in every case and the code wrong in every case** — the empirical argument for "plans should specify abstractions, codebases should specify call sites." 0/7 phases needed structural rework, 3/7 needed signature touch-up — the right ratio for a healthy plan.

### Phase 8 — Post-Review Hardening (`docs/completions/hermes-adapter-and-fly-wiring-phase-8.md`)

**Off-plan, driven by code-review against Phases 1–7.** Phases 1–7 landed M3 functionally complete and architecturally clean against the ten blueprint §11 / stack.md §11 rules; a thorough multi-angle review (the user-driven 9/10 quality gate before push) surfaced six concrete improvements: one **load-bearing test gap** (the four pure functions in `fly.go` had zero functional tests backing them), one **load-bearing semantic gap** (`Spawn` orphaned Fly apps on partial-failure paths between `CreateApp` and `Launch`), and four polish items. **Phase 8 adds no new wire surface, no schema, no proto, no domain packages** — it tightens the existing M3 deliverables against their own contracts.

**`internal/deploy/target_test.go` rewrote from 11 LOC of compile-time assertions to 110 LOC with 24 functional sub-tests.** `TestValidateImageRef` (7 cases — including the load-bearing `contains_sha256_but_no_@` canary against §11.2 relaxation; a future "let's just check for `sha256` somewhere" cleanup would silently bypass §11.2 at the call site without tripping any check, and this test is the canary against exactly that). `TestMapFlyState` (10 cases — including the `unknown-future-state` policy pin: any state Fly emits that we don't recognise is `HealthFailed`, not `HealthUnknown`; "fail closed" is correct for a deploy target but is the kind of decision that gets flipped in a "let's not be too aggressive" cleanup; the test name documents the *intent*). `TestAppNameFor` (3 sub-tests — UUID-passthrough, deterministic-hashing, no-collision; the function does double-duty and the callers will rely on *both* properties without distinguishing them at the call site). `TestParseExternalRef` (4 cases). The package flips from `ok ... [no tests to run]` to `ok ... 0.165s` running real assertion bodies.

**`Spawn`'s rollback uses the named-return + `defer` + `if err != nil` idiom.** Three load-bearing properties. (1) **The defer auto-disarms on success** because `err == nil` at that point — no flag variable, no goroutine, no extra branches at every error site. (2) **Cleanup uses `context.WithTimeout(context.Background(), 30s)`, not the caller's ctx.** Non-negotiable: when Spawn fails *because the caller's ctx was canceled* (deadline exceeded, client disconnected mid-spawn, parent-context cleanup), reusing the canceled context would cancel the cleanup too — orphaning the very app the rollback exists to delete. (3) **Cleanup failures `slog.Warn` but don't override the original `err`.** The caller already has a structured spawn-time error; replacing it would obscure root cause. The slog line names both errors (`spawn_err`, `cleanup_err`) and the affected app, so a single grep against the JSON log identifies every orphan candidate. **Spawn is now resource-idempotent at the failure boundary** — M4's spawn-flow handler is a thin caller, no compensation logic.

**`Health` rejects `>1 machine` with a structured error citing blueprint §8.** Pre-Phase-8 `Health` enforced nothing on the *read* path — a Fly app with two machines (created via M4 ops error, manual `flyctl machine clone`, retry-storm bug) would have its health reported as machine[0]'s state regardless of machine[1]'s state. Post-Phase-8 returns `(HealthUnknown, error)` — not silently OK, not silently FAILED, but *loudly UNKNOWN*. Choice of `HealthUnknown` over `HealthFailed` is intentional: the AgentInstance state machine can interpret "unknown — operator review needed" differently from "failed — destroy and respawn." **The v1 invariant (one AgentInstance = one Fly app = one Fly machine) is now self-validating in both directions** — Spawn enforces on creation, Health on read.

**`keysOf` sorting is a quiet-but-real determinism fix for the boot-log runbook.** Pre-Phase-8 the log line `kinds=fly,local,aws` was the *intent*, but Go intentionally randomizes `for k := range m` iteration; actual emitted value rotated across permutations. Phase 6's completion doc had treated `"kinds=fly,local,aws"` as a stable runbook artefact — operators grepping boot logs to confirm "all three deploy targets registered" had a value that wasn't actually stable. `sort.Strings(out)` makes the log line deterministic across boots: always `"aws,fly,local"`, in alphabetical order. **Trivial change, real correctness gain for a runbook contract.**

**`_ = adaptersSvc` deletion follows CLAUDE.md's "delete unused code completely, don't keepalive" rule.** Phase 6 had introduced the keepalive with a comment naming M4 as the consumer; Phase 8 deletes it (and the `adapters` import) because "M4 will need this in N weeks" is exactly the kind of speculative-future-requirement CLAUDE.md flags as a defect. **The two-line addition in M4 (re-add `adapters` import + `adaptersSvc := adapters.NewService(queries)`) is cleaner than carrying a keepalive comment that ages.** Sign of healthy churn: removed lines beat added lines on a code-quality pass.

**The `// SAFETY:` carve-out in `cmd/smoke-deploy/main.go` documents an intentional rule deviation rather than silencing it.** Pre-Phase-8 the binary read `CORELLIA_SMOKE_API_KEY` and `CORELLIA_HERMES_ADAPTER` directly via `os.Getenv`, technically violating blueprint §11.8 ("all env vars read through `internal/config/`"). Phase 7's completion doc *implicitly* argued this was operator-tooling and exempt from the rule's spirit; Phase 8 makes the argument *explicit* with an 8-line comment block naming §11.8, naming the carve-out (smoke-only knobs shouldn't ride the prod-server fail-fast-at-boot path), and naming the alternative (adding them to `Config` would put smoke-test plumbing in every production process's required-env list). **Documenting an exception is not the same as silencing a rule.**

**Quality scorecard moved from 8.5/10 → 9/10.** `internal/deploy/fly.go` 9 → 9.5 (rollback + invariant guard); `target_test.go` 6 → 9 (24 sub-tests pinning 4 pure-function contracts); `cmd/api/main.go` 8 → 9 (keepalive deleted, deterministic log); `cmd/smoke-deploy/main.go` 8 → 9 (deviation documented at call site). All ten blueprint §11 / stack.md §11 architecture rules pass at the post-Phase-8 commit point.

### Behavior change (known)

- **`Spawn` no longer leaves orphan Fly apps on partial failure.** Pre-Phase-8 callers that retried after a `Spawn` error needed to call `Destroy` themselves to clean up; post-Phase-8 the cleanup happens inside `Spawn` before the error returns. M4's spawn-flow handler is a thin caller — no compensation logic. **Operationally: any Fly account that already has orphan apps from pre-Phase-8 testing needs one-time manual cleanup via `flyctl apps destroy <name>`** (the rollback only catches future failures).
- **`Health` returns an error for apps with >1 machine.** Any pre-existing app that has 2 machines surfaces as `(HealthUnknown, error)` to the caller rather than silently reporting machine[0]'s state. M4's spawn-flow status updater needs to handle this `HealthUnknown + err` shape — most likely by surfacing it to the operator as "manual reconciliation needed" rather than auto-correcting.
- **`cmd/api`'s boot log line `"deploy targets initialised"` always emits `kinds=aws,fly,local` in alphabetical order.** Operators grepping for substring `"fly"` are unaffected; anyone who pinned the *exact* string `"fly,local,aws"` (the pre-sort accidental-default) needs to update.
- **Live wire surface gained zero new routes.** Phase 6 wired `httpsrv.Deps.DeployTargets` but no handler reads it in M3; M4 is the first reader. The HTTP surface is identical to M2 (`POST /corellia.v1.UsersService/GetCurrentUser`, `POST /corellia.v1.OrganizationsService/CreateOrganization`, `POST /corellia.v1.AgentsService/ListAgentTemplates`).
- **First production caller of any Fly-touching code path.** Every `cmd/api` boot now constructs a real `*flaps.Client` against Fly's HTTPS API (`flaps.NewWithOptions`); the binary fails fast at boot if `FLY_API_TOKEN` is missing/malformed. Note: `flaps.NewWithOptions` does *not* make a network call to verify the token is live — the first failure point on a stale/invalid token is `Spawn` (or `Stop` / `Destroy` / `Health`), not boot.
- **Eight new product-feature artefacts committed** under domain-meaningful paths: `adapters/hermes/{Dockerfile, entrypoint.sh, README.md, .dockerignore, smoke.sh}`; `backend/migrations/20260426120000_adapter_image_ref_backfill.sql`; `backend/internal/deploy/{target.go, fly.go, stubs.go, target_test.go}`; `backend/internal/adapters/service.go` (extended); `backend/queries/harness_adapters.sql` (extended); `backend/cmd/smoke-deploy/main.go`; `backend/cmd/api/main.go` (extended); `backend/internal/httpsrv/server.go` (extended). Plus the regenerated trees (Go DB layer for the new query + sqlc models type-flip).

### Phase 1 — Schema migration + seed (`docs/completions/agent-catalog-phase-1-completion.md`)

**Migration packaging.** `20260425170000_agent_catalog.sql` — schema + seed in one transaction. Bundling means a partially-applied state is impossible; goose handles the per-file atomicity implicitly. The seed is *part of the schema's correctness contract* — without the Hermes seed, every read query returns empty and the entire `/agents` page is a debug surface. Plan decision 2. (This is a deliberate exception to the "schema migrations don't carry data" rule: the seed here is what makes the schema meaningful on first apply, not bulk fixture data.)

**Two CHECK constraints encode architectural rules.** `CHECK (upstream_image_digest LIKE 'sha256:%')` rejects mutable-tag writes at the column level — encodes blueprint §11.2 in the database, not just in code. `CHECK (source IN ('hand_written', 'generated'))` enumerates the v1 sources without `CREATE TYPE` ceremony (no separate down-block step, no migration ordering for type drops); v2's generated-adapter pipeline lands without a schema change. Decisions 4 + 5.

**Idempotency.** The harness seed uses `ON CONFLICT (harness_name) DO NOTHING` because the table has a unique index. The template seed has no natural unique key (post-v1 you can have multiple templates per adapter), so it uses `WHERE NOT EXISTS (SELECT 1 FROM agent_templates t WHERE t.harness_adapter_id = ha.id)` — the v1 invariant ("one template per adapter") encoded in the seed without requiring a unique index that would have to be dropped later. Defence-in-depth across `goose down` + `up` cycles.

**Image digest capture.** Pre-work used `crane digest ghcr.io/nousresearch/hermes-agent:<version>` to resolve the version tag (not `:latest`) to its multi-arch manifest-list digest. The audit-comment block above the seed `INSERT` records `resolved_tag`, `digest`, `captured_at` — a future reader can verify "yes, that digest was the Nous v0.x.y release as of 2026-04-25." Plan decision 22's three governance moves: (a) starting from a version tag means we know *which release* we're pinning; (b) `crane digest` returns the manifest-list digest (correct granularity for an orchestrator that may run multiple host architectures); (c) the SQL comment makes the pin auditable.

**Down block.** Strict LIFO with `IF EXISTS` guards: `DROP TABLE IF EXISTS agent_templates;` then `DROP TABLE IF EXISTS harness_adapters;`. Re-runnable. Tested with a `goose down && goose up` cycle that produced the same end state; the CHECK constraint was hand-verified against a `INSERT ... VALUES ('test', 'latest', ...)` which raised the expected constraint violation, then rolled back.

**`adapter_image_ref` is `TEXT NULL` in M2.** M3 fills it and tightens the column to `NOT NULL` in a follow-up migration. Plan decision 3 — making it `NOT NULL` in M2 would force an empty-string placeholder that fails §11.2 in spirit ("never a tag" is meant to forbid mutable refs; an empty string is worse). Nullable + a comment saying "set by M3" is the honest schema. The blast radius of the nullability is bounded: M2 reads do not consume the column at all (decision 11's narrow proto projection), and `internal/adapters` is an unreaderable package today (decision 21).

### Phase 2 — sqlc queries + regen (`docs/completions/agent-catalog-phase-2-completion.md`)

**Two new query files, deliberately one-method-each.** `backend/queries/agent_templates.sql` defines `ListAgentTemplates :many`; `backend/queries/harness_adapters.sql` defines `GetHarnessAdapterByID :one`. Convention is one `.sql` file per table. `harness_adapters.sql` exists so the package compiles with non-empty surface area and M3's `UpdateAdapterImageRef` extends rather than scaffolds (decision 15). Plan-level argument: M3 reads "add `UpdateImageRef`" instead of "create the package, decide on the shape, then add `UpdateImageRef`" — saves a round of architectural relitigation in two months.

**Narrow projection, not `SELECT *`.** `ListAgentTemplates`'s SQL projects exactly four columns (`id, name, description, default_config`) — sqlc's generated row type matches the selected columns, so `db.ListAgentTemplatesRow` physically cannot leak `harness_adapter_id`, `created_by_user_id`, or timestamps to the catalog service. **Column projection is API design, not optimization.** This is the cheapest possible authorization-through-typing technique in the codebase: a future caller that needs the omitted fields has to write a new query, which forces a code review of the privacy-vs-functionality tradeoff. Decision 11's plan-level rationale plays out at the SQL layer.

**`default_config` returns as `[]byte` via pgx's default `JSONB` mapping.** No FE consumer in M2 (the proto omits the field per decision 11); the row type carries it for M4's deploy modal to either parse server-side or marshal to a JSON string in the proto.

**Generated artefacts.** Four files under `backend/internal/db/`:
- `agent_templates.sql.go` — new; contains `ListAgentTemplates(ctx) ([]ListAgentTemplatesRow, error)` + the narrow row type.
- `harness_adapters.sql.go` — new; contains `GetHarnessAdapterByID(ctx, id) (HarnessAdapter, error)` + the standard `HarnessAdapter` row type (this query projects `*`, so the row type matches `models.go`'s `HarnessAdapter` struct).
- `models.go` — gains `HarnessAdapter` + `AgentTemplate` row structs + `ListAgentTemplatesRow` (the narrow projection's struct, distinct from `AgentTemplate`).
- `querier.go` — gains `GetHarnessAdapterByID` + `ListAgentTemplates` on the `Querier` interface.

**`emit_pointers_for_null_types: true`** (the global flag from 0.1.0's `sqlc.yaml`) means `agent_templates.created_by_user_id` would be `*uuid.UUID` if we projected it — but we don't, so the question is moot. Worth flagging because it's the kind of detail that re-emerges every time a new nullable column lands; the global flag handles it without per-column overrides.

### Phase 3 — Proto + buf regen (`docs/completions/agent-catalog-phase-3-completion.md`)

**`shared/proto/corellia/v1/agents.proto`** — new file. One service (`AgentsService`), one RPC (`ListAgentTemplates`), three messages (`ListAgentTemplatesRequest` empty, `ListAgentTemplatesResponse` with `repeated AgentTemplate templates = 1`, `AgentTemplate` with three string fields). One service per file convention from `stack.md` §3. The `corellia.v1` package namespace is permanent — breaking changes go to `v2`, never silent edits to `v1`.

**Generated artefacts (committed).** `backend/internal/gen/corellia/v1/agents.pb.go`; `backend/internal/gen/corellia/v1/corelliav1connect/agents.connect.go` (provides `AgentsServiceHandler` interface and `NewAgentsServiceHandler` factory); `frontend/src/gen/corellia/v1/agents_pb.ts` (exports `AgentsService` descriptor and the message classes with `Message<"corellia.v1.AgentTemplate">` brand for type-narrowing). ### Phase 4 — Backend domain + handler (`docs/completions/agent-catalog-phase-4-completion.md`)

**Two new domain packages.** `internal/adapters/` (35 LOC, one file) and `internal/agents/` (51 LOC, one file). Both mirror the established `users/` + `organizations/` shape: private query interface (`adapterQueries` / `templateQueries`) listing only the methods the service touches; sentinel errors at the top of the file (`adapters.ErrNotFound`, `agents.ErrNotFound`); typed-row → domain-or-proto conversion at the service layer; constructor-injected dependencies. Plan decisions 13–16.

**`adapters.Service.Get` exists with no caller.** Decision 13's plan-level rationale: `M3 (adapters)` and `M4 (agents)` extend rather than scaffold. Building both packages now (even when one is empty) means M3's plan reads "add `UpdateImageRef`" instead of "create the package, decide on the shape, then add `UpdateImageRef`." Saves a round of architectural relitigation in two months. The package compiles, the type is in scope, no instance lives at runtime — the cleanest expression of "scaffold the package without scaffolding the runtime."

**`agents.Service.ListAgentTemplates`'s `make([]*…, 0, len(rows))` line is load-bearing.** sqlc's `ListAgentTemplates` returns `var items []ListAgentTemplatesRow` — which is `nil` when zero rows. If we passed that through unchanged, `corelliav1.ListAgentTemplatesResponse{Templates: nil}` marshals to `{"templates":null}` over Connect's default JSON codec. The FE's `data.templates.length` would crash with `Cannot read properties of null`. The `make(..., 0, len)` guarantees a non-nil empty slice → marshals as `{"templates":[]}` → FE gets a real array even on zero rows. This kind of nil-versus-empty distinction comes up *constantly* in Go-to-typed-frontend wire boundaries; the fix is always "constructor with explicit zero capacity," and Phase 6's `_Empty` test pins it. The code comment in the service explains the choice in-line so a future code-cleanup pass doesn't regress it.

**Why `ErrNotFound` is declared but unused in M2.** The handler's `agentsErrToConnect` switch has a `case errors.Is(err, agents.ErrNotFound)` arm that no caller in M2 surfaces. Strict reading of CLAUDE.md ("don't design for hypothetical future requirements") would forbid this. The exception case: when the *next named milestone* (M4, with `GetAgentTemplate(id)`) is named in the codebase comments, has a plan doc, and produces a concrete branch in the dependency graph, "future" stops being hypothetical and becomes "named-but-not-yet-built." The 4 lines mean M4's PR is one fewer line; M4's plan doc is one fewer review concern; the contract surface is established at the *first* RPC, not retrofitted at the second.

**Handler error mapping.** `agentsErrToConnect` switch with two arms: `ErrNotFound` → `connect.CodeNotFound`; `default` → `slog.Error("agents handler: unexpected error", ...)` + `connect.NewError(connect.CodeInternal, errors.New("internal error"))`. The redaction is non-negotiable per the 0.2.5 post-review hardening — pgx/driver errors must not leak across the wire surface. Operator visibility goes through `slog`; client visibility shows the redacted `"internal error"` string only. Same shape `users_handler.go:44-53` and `organizations_handler.go:45-48` use; three handlers, one architectural pattern.

**Wiring deltas.** `httpsrv/server.go` (+4 / -0): `AgentsHandler` field on `Deps` between `OrganizationsHandler` and `AllowedOrigin`, plus two lines mounting the new service inside the existing `r.Group(auth.Middleware(...))` block (decisions 19 + 20 — catalog is authenticated; alphabetical-ish-by-domain ordering). `cmd/api/main.go` (+3 / -0): import + `agentsSvc := agents.NewService(queries)` + `Deps{...}` field assignment. `adapters.NewService(...)` is *deliberately* not wired — decision 21; M3 wires when `UpdateAdapterImageRef` becomes a real reader.

**Structural typing collapses the wiring.** `*db.Queries` (returned by `db.New(pool)`) structurally satisfies *all four* per-service interfaces (`userQueries`, `orgQueries`, `templateQueries`, `adapterQueries`) without anyone editing `*db.Queries`. Adding a fifth service is a five-line file in `internal/<new>/service.go` plus a constructor call in `main.go`; no central registry to update, no shared type to widen. This is what CLAUDE.md's "Don't add features, refactor, or introduce abstractions beyond what the task requires" looks like in practice — the abstraction (private per-service interface) is *required* (test-seam need + tight dependency surface), not speculative. The fact that it also enables zero-coupling-cost service expansion is a consequence, not a goal.

### Phase 5 — Frontend `/agents` page (`docs/completions/agent-catalog-phase-5-completion.md`)

### Why each fix

#### Why memoize the provider value via a sub-component, not inline

`useMemo` is a hook — it can only run inside a function-component body. The layout's render branches (`loading`, `not-provisioned`, `error`, `ready`) early-return; placing `useMemo` at the top of the layout would run it on every render including the redirect/error paths where `user`/`org` aren't yet defined. Extracting the ready-state render to a `<ReadyChrome>` sub-component scopes the hook to the only branch where the inputs exist, stays clean against React's rules-of-hooks ESLint rule, and keeps the parent layout's branch logic flat. Net diff: +13 lines (the sub-component declaration), -3 (the inlined render block).

#### Why `replace` everywhere, not just sign-in

The codebase's documented rule (Phase 2 doc): *`replace` for forced navigation that shouldn't be reversible by back-button; `push` for user-initiated nav.* Sign-in is the canonical forced one-shot — once the redirect lands on `/dashboard`, there's no legitimate "I want to go back to the sign-in form" reason. The previous `push` was almost certainly leftover from the original 0.2.0 scaffold (the convention was tightened in Phase 2). Aligning the entry-point of the auth flow with the rest of it is consistency-as-defect-fix.

#### Why a throw in `client.ts`, not a typed Result

The helper has exactly one consumer family: components that need a Supabase browser client at user-interaction time. Threading a `Result<Client, ConfigError>` through every call site would be an order of magnitude more code for a failure mode that is purely a *configuration* error (env vars missing) — the right action is "crash loudly with a useful message," not "let every caller branch on it." Same fail-loud rationale that drove the JWKS verifier's `NoErrorReturnFirstHTTPReq=false` decision in 0.2.6 and `useUser()`'s out-of-provider throw in 0.3.0 Phase 5. Three load-bearing pieces of FE/BE infrastructure now share the convention: misconfiguration is a programming error, not a runtime branch.

#### Why `title: { absolute }` for the wizard, not the template default

The wizard's title is `"Welcome to Corellia"`. With the template applied, the bare title `"Welcome"` would render as `"Welcome — Corellia"` — grammatically valid but reads as a tagline rather than a route name. The `{ absolute: ... }` form bypasses the template specifically for this one route. Cost: one extra metadata key. Benefit: the full sentence "Welcome to Corellia" reads naturally to a first-time signed-in user, who is the entire audience for this title.

#### Why the `proxy` rename can't be skipped indefinitely

Two reasons beyond the obvious "deprecation warnings are noise":

1. **Next.js's `proxy` API is the forward path.** The convention rename is a precursor to capabilities that don't exist on the legacy `middleware` API (the deprecation message links to `nextjs.org/docs/messages/middleware-to-proxy`, which itself points at the new feature surface). Migrating now means we adopt new capabilities cleanly when we want them; deferring means a forced migration plus a feature adoption in one shot.
2. **The deferral was already two milestones long.** Phase 1's findings logged it; Phase 6's findings re-logged it; this entry retires it. Three log entries in two months is the threshold at which "deferred polish" becomes "ignored debt."

The migration's blast radius is genuinely small (two-line file, no consumers because Next.js auto-discovers the export), so doing it now is the cheap call.

### Phase 2 — Onboarding wizard at `/onboarding`

State machine `loading | ready | not-provisioned | error` plus a `submitting` flag *inside* the `ready` branch (not as its own state — keeping the form data in scope while the request is in flight is what enables the form re-enable on error). `react-hook-form`'s `defaultValues` are read once at form construction; by the time `getOrganization` resolves, the form already exists with empty defaults. `form.reset({ name: "", orgName: org.name })` re-initialises without flicker (no validation pass) and keeps `formState.isDirty` semantics intact — preferred over `setValue` (which would mark the field as dirty by default).

Submit fires `Promise.all([updateCurrentUserName, updateOrganizationName])`. The two updates are independent (`public.users` vs `public.organizations`); `Promise.all` halves wall-clock latency. **Acknowledged failure mode:** if one update lands and the other rejects, the user is in a half-onboarded state. Re-entering the wizard re-fetches and pre-fills with whichever value already persisted; the layout's `user.name?.trim()` predicate stays false if the failing half kept it empty, so the user sees the form again with the half that succeeded already saved. They finish the unfinished half and continue. Accepted; not a blocker.

`router.replace` for every navigation (forced redirects shouldn't be reversible by back-button); `signOut` available on every render branch (the wizard is the first route a freshly-provisioned user sees, and an unprovisioned user is otherwise trapped — every navigation ends back here). **Sibling `layout.tsx`** for the `metadata.title = "Welcome to Corellia"` export — recurring pattern across the App Router whenever a leaf page is `"use client"` but still needs static metadata (recurs at `/dashboard` in Phase 5).

`aria-invalid={!!form.formState.errors.name}` is wired onto each `<Input>` so the destructive-coloured error ring lights up automatically — closest equivalent to a `FormField`/`FormItem` wrapping with the deps available. Asymmetric defaulting on the two fields: org name pre-filled (reveals what the trigger did, lets the user keep or override); user name empty (no good default — `email[0..@]` is a username, not a display name).

## 0.2.6 — Auth Migration: HS256 Shared Secret → ES256 / JWKS (2026-04-25)

Migrated backend JWT validation from the legacy HS256 / `SUPABASE_JWT_SECRET` shared-secret path scaffolded in 0.1.0 to the current asymmetric ES256 / JWKS path Supabase issues today. Backend now fetches Supabase's JWKS document once at boot from `$SUPABASE_URL/auth/v1/.well-known/jwks.json`, caches it in memory, refreshes it hourly in the background plus immediately on any unknown `kid` (rate-limited to one refetch per 5 min), and validates incoming access tokens offline against the cached P-256 public keys. `SUPABASE_JWT_SECRET` is removed from `config.Config`, `.env.example`, and `backend/.env`. Frontend is unaffected — the access token is opaque to the client. Plan: `docs/executing/auth-es256-migration.md`. Six phase-completion docs under `docs/completions/auth-es256-migration-phase{1..6}-completion.md`.

### Index

- **New dependency.** `github.com/MicahParks/keyfunc/v3@v3.8.0` direct (canonical JWKS adapter for `golang-jwt/jwt/v5`); `github.com/MicahParks/jwkset@v0.11.0` and `golang.org/x/time@v0.9.0` indirect.

### Phase 1 — Add JWKS library dependency

`go get github.com/MicahParks/keyfunc/v3@v3.8.0` from `backend/`. Pulled `jwkset` + `x/time` transitively. Skipped the plan's `go mod tidy` step deliberately — tidy at this phase boundary, before any source file imports keyfunc, would have *removed* the dep entirely (Go's tidy prunes unimported requires). ### Phase 3 — Auth middleware rewrite

Added `backend/internal/auth/jwks.go` (40 lines, single struct + constructor + method) and rewrote `backend/internal/auth/middleware.go` for ES256 + JWKS. `claims.go` untouched — `AuthClaims` shape is unchanged across the migration by design (decision 8 in the plan). Two real semantic changes in the middleware: (1) **key source** flipped from `func(t *jwt.Token) (any, error) { return []byte(jwtSecret), nil }` (returns the same HMAC secret regardless of token) to `verifier.Keyfunc()` (looks up the public key by the token's `kid` header); (2) **algorithm enforcement** added via `jwt.WithValidMethods([]string{"ES256"})` — closes the classic algorithm-confusion attack where an attacker signs an HS256 token using the *public* key as the HMAC secret and the verifier accepts it because `jwt/v5` dispatches on the token's claimed `alg` header. *Where:* `backend/internal/auth/jwks.go` (new), `backend/internal/auth/middleware.go` (rewrite). *Why this is the structural shape rather than the plan's verbatim shape:* keyfunc v3.8.0's `NewDefaultOverrideCtx` collapses the plan's three-step manual sequence (`NewDefaultHTTPClient` → `KeyReadAll` → `keyfunc.New`) into one call, picking up keyfunc's defaults the plan listed aspirationally (1h refresh, 5-min rate-limited unknown-kid refetch). The `NoErrorReturnFirstHTTPReq: &false` + `HTTPTimeout: 10*time.Second` overrides invert keyfunc's swallow-first-fetch-failure default for our specific use case (a misconfigured `SUPABASE_URL` should fail loud at boot, not produce a stream of 401s after a stale empty cache survives).

**Architectural notes worth preserving.** (1) `JWKSVerifier` is a struct wrapper around `keyfunc.Keyfunc`, not a type alias — insulates call sites in `httpsrv` and `main.go` from the inevitable v3→v4 churn (the plan explicitly flagged keyfunc API shifts between minor versions), and keeps the keyfunc transitive-vocabulary out of those packages, respecting blueprint rule §11.6 in spirit. (2) The ES256 whitelist is hardcoded, not config-driven — Supabase has a single canonical signing algorithm per project; making this configurable would be surface-area with no legitimate variation *and* a footgun, since adding HS256 back to the whitelist re-opens the algorithm-confusion attack. (3) `keyfunc.UseWhitelist` (filters JWKS entries by JWK `use`) was deliberately *not* added — it's the wrong knob; algorithm-confusion defence is an `alg`-vs-`use` distinction and lives at parse time via `jwt.WithValidMethods`, not at key-time via JWK filtering. Adding `UseWhitelist` would couple us tightly to Supabase publishing `"use": "sig"` forever.

### Phase 4 — `main.go` + `httpsrv` wiring

Added `AuthVerifier *auth.JWKSVerifier` to `httpsrv.Deps` between `Config` and `UsersHandler` — a rough request-path ordering ("config → auth infra → app handlers → CORS policy") that future cross-cutting concerns (rate limiter, tracer, metrics sink) slot into naturally. Flipped the router's auth middleware call site from `auth.Middleware(d.Config.SupabaseJWTSecret)` to `auth.Middleware(d.AuthVerifier)` at `httpsrv/server.go:34`. In `cmd/api/main.go`, derived the JWKS URL via `strings.TrimRight(cfg.SupabaseURL, "/") + "/auth/v1/.well-known/jwks.json"` (the trim defends against an operator pasting their dashboard URL with a trailing slash — quiet bug-class closed), constructed the verifier eagerly via `auth.NewJWKSVerifier(ctx, jwksURL)`, used `slog.Error`+`os.Exit(1)` on failure (matches the `db.NewPool` boot-error idiom; mixing `log.Fatalf` and `slog.Error` would produce two log shapes on stderr), and added a single `slog.Info("jwks initialised", "url", jwksURL)` line — keyfunc only logs on refresh *failures*, so without this line a successful boot is silent and a silent failure is indistinguishable from a silent success. *Where:* `backend/cmd/api/main.go`, `backend/internal/httpsrv/server.go`. *Why a `Deps` field rather than reading the verifier out of `Config`:* `config.Config` is a typed env-var snapshot — its fields come from `os.Getenv` and are strings/ints. The verifier is runtime-constructed infrastructure, parallel to `db.Pool`, not a parsed config value. Same reasoning kept `db.Pool` out of `Config` originally.

### Behavior change (known)

**Breaking config change.** Operators and CI environments that previously set `SUPABASE_JWT_SECRET` must remove it; nothing reads the variable any more. The Go binary no longer panics at config-load if `SUPABASE_JWT_SECRET` is unset (it would have, before — `caarlos0/env` honored the `required` tag). Conversely, the binary will now fail loudly at *boot* (after config-load, before listening) if `SUPABASE_URL` is missing or unreachable, with `auth: initial JWKS fetch from <url>: <err>` — same fail-fast posture, different gate.

**No request-path behavior change after a successful boot.** A valid Supabase access token validates and the request flows through to the domain layer; an invalid one returns 401. The wire-level error strings (`"missing bearer token"`, `"invalid token"`, `"invalid sub claim"`) are unchanged. `AuthClaims` shape is unchanged. Downstream services see no difference.

### Supersedes

- **0.1.0 "Auth middleware (HS256)."** The "shared secret + HMAC signature check" model documented at scaffolding time is wholly retired. Rationale upgrades from "Supabase issues HS256 tokens, validate offline with the secret" to "Supabase issues ES256 tokens, validate offline with the cached public key — server-side rotations propagate within an hour without a backend restart." The post-migration posture is strictly stronger: asymmetric crypto removes the attack surface where a leak of `SUPABASE_JWT_SECRET` from operator infrastructure would have let an attacker mint arbitrary tokens — there is no symmetric secret to leak now.

### Phase 1 — Schema migration (`docs/completions/auth-user-provisioning-phase-1.md`)

**Provisioning trigger.** `public.handle_new_auth_user()` runs `AFTER INSERT ON auth.users` and atomically creates a fresh `public.organizations` row (named `split_part(NEW.email, '@', 1) || '''s Workspace'`) plus the matching `public.users` row with `role = 'admin'`. Runs in the same transaction as the auth insert, so either both rows land or neither does — no dangling auth users without a public record. *Where:* `backend/migrations/20260424140000_auth_user_provisioning.sql`. *Why:* implements decision 1 of the plan (Postgres trigger over Go-code provisioning) — atomic with the auth insert, fires regardless of entry point (dashboard, future signup flow, admin API), and is Supabase's canonical shape. Goose `+goose StatementBegin/End` fences are required for plpgsql function bodies — without them goose splits on `;` mid-function and the migration fails silently.

**Cleanup trigger — Pattern-A-correct now, Pattern-C-safe later.** `public.handle_deleted_auth_user()` runs `AFTER DELETE ON auth.users`: removes the `public.users` row, then conditionally removes the org *iff* `NOT EXISTS (SELECT 1 FROM public.users WHERE org_id = old_org_id)`. Today (Pattern A: one admin per org) the condition is always true, so cleanup is total. When invitations land (Pattern C: multi-member orgs) the condition becomes *sometimes* true and multi-member orgs survive the deletion of any one member — the trigger doesn't change between the two patterns. *Where:* same migration. *Why:* implements decision 10 of the plan ("provision and delete are two sides of the same coin"). Picking the right cleanup shape now — instead of writing a Pattern-A-only DELETE that needs rewriting at v2 — costs zero extra lines and one extra subquery.

**`SECURITY DEFINER` + locked `search_path` is non-negotiable.** Both functions use `SECURITY DEFINER` (so they run with the owner's privileges — the postgres superuser, since goose applied via `DATABASE_URL_DIRECT`) and `SET search_path = public, pg_temp` (so a malicious schema entry on the search path can't shadow `public.organizations` and execute under elevated privileges). Without the search_path lock-down, `SECURITY DEFINER` is a classic privilege-escalation target; both together are the canonical Postgres hardening. *Where:* same migration. *Why:* the function is invoked in `supabase_auth_admin`'s context (no grant on `public`) but writes `public.*`. `SECURITY DEFINER` is the only mechanism that works here, and the search_path pin closes its known foot-gun.

**`ALTER TABLE public.users ADD COLUMN name TEXT NULL`** — primer for the onboarding wizard follow-up plan. Nullable so unprovisioned-via-dashboard users don't need a name to exist; the wizard (next plan) prompts on first login. *Why:* CLAUDE.md flags half-finished implementations — adding `name` now without an `UpdateCurrentUserName` RPC would be exactly that, so both land in this pass per plan decision 15.

**Down block is strict LIFO with `IF EXISTS` guards** — drops triggers, then functions, then the column. Re-runnable. Data in `public.users` / `public.organizations` is *not* touched (those are data, not schema artefacts).

**Trigger semantics under the Supabase auth role were not exercised in Phase 1.** What's verified: function bodies compile, triggers are installed on the right events. What is not: that `SECURITY DEFINER` actually grants `INSERT` privileges on `public.organizations` + `public.users` under `supabase_auth_admin`'s execution context, and that the search_path lock-down doesn't accidentally hide a needed object. Picked up in the Phase 4 manual E2E runbook (E2E-1 / E2E-3) and listed below as the operator-side verification still needed before first non-localhost deploy.

ErrNotProvisioned   →  PermissionDenied  (FE: render the provisioning panel)
ErrForbidden (org)  →  PermissionDenied  (FE future work distinguishes)
ErrNotFound (org)   →  NotFound          (FE: 404 state)
<anything else>     →  Internal          (FE: surface message)

### Post-Review Hardening — Internal error redaction (`docs/completions/auth-user-provisioning-post-review-hardening.md`)

**Both handler error switches now redact unknown errors at the wire boundary.** `default:` arm in `httpsrv/users_handler.go`'s `toConnectErr` and `organizations_handler.go`'s `orgErrToConnect` was returning `connect.NewError(connect.CodeInternal, err)` — and connect-go propagates the wrapped error's `.Error()` string into the response body the client receives. So a `pgx`-level error like `failed to connect: dial tcp [2406:da18:...]:5432: i/o timeout` or `ERROR: relation "users" does not exist (SQLSTATE 42P01)` would land in the browser. New shape: `slog.Error("...", "err", err); return connect.NewError(connect.CodeInternal, errors.New("internal error"))`. The full diagnostic stays in the server logs (already JSON-formatted via `slog.NewJSONHandler` in `cmd/api/main.go:22`); the client receives only `{"code":"internal","message":"internal error"}`. *Where:* `backend/internal/httpsrv/{users_handler.go, organizations_handler.go}`. *Why:* surfaced by the post-completion review of Phases 1–4 as the only finding treated as pre-prod blocking — leaks DB topology, schema details, and infrastructure shape to anyone who can produce a 500 (including unauthenticated traffic that gets past the middleware due to a future bug). Sentinel arms are unchanged: their messages are part of the public contract (the FE branches on the Connect *code*, not the string), so `users.ErrNotProvisioned` and friends still flow through verbatim.

**Why redaction lives at the handler boundary, not the service.** The temptation when fixing an error-leak is to "sanitize at the source" — wrap every service-layer error in a clean message. That's the wrong layer: the service doesn't know whether its caller is a Connect handler (where leaks are bad), an internal cron job (where the raw error is what you want in logs), or a future test harness. Redaction is fundamentally a *transport* concern — about what crosses the network boundary — and the handler is the only place that knows that boundary exists. Same architectural reasoning as CLAUDE.md rule §11.9 ("business logic never in handlers"): each layer owns the concerns specific to its position.

## 0.2.4 — `DATABASE_URL` Canonicalized to Direct Connection (2026-04-24)

Rewrote `DATABASE_URL`'s documented connection mode from Supabase **Session Pooler** (`*.pooler.supabase.com:5432`) to **Direct Connection** (`db.<ref>.supabase.co:5432`). Both `DATABASE_URL` and `DATABASE_URL_DIRECT` now point at the same Direct host — the split is role-and-lifecycle, not host-vs-host. Motivated by Supabase's own UI copy ("Direct Connection: ideal for applications with persistent and long-lived connections, such as those running on virtual machines or long-standing containers"), which describes a Go+Fly single-binary orchestrator exactly. Supersedes 0.1.0's "Session Pooler for app, Direct for migrations" framing and re-grounds the rationale on the missing architectural fact: **`pgxpool` is an in-process transaction pooler**, so an external pooler on the wire is redundant.

### Index

- **Docs updated:** `CLAUDE.md` §Database connection (full rewrite with the pgxpool-as-transaction-pooler rationale + ceiling math) and §Migrations heading; `docs/stack.md` §6 migrations clause + new §8 "Database URLs — both Direct Connection, split by role" subsection + `DATABASE_URL` table row; `.env.example` header comment on both URLs (example URL flipped to Direct); `docs/blueprints/codegen-cheatsheet.md` quick rule; `docs/completions/frontend-scaffold-completion.md` pending-work reference.
- **Code comment updated:** `backend/internal/config/config.go` — `DatabaseURL` godoc rewritten (now names Direct + pgxpool semantics + IPv4 fallback + Transaction Pooler red line).
- **No runtime code changed.** `pgxpool.Config` values unchanged (`MaxConns=10`, `MinConns=2`, lifetime + idle + health-check); connection URL shape is indifferent to the driver; only the documented convention + the populated value in `backend/.env` changes.
- **Session Pooler reclassified as the IPv4 fallback**, not the primary. Swap is drop-in — different URL, same driver, same `pgxpool` — so per-developer variation costs nothing and doesn't leak into committed config.
- **Transaction Pooler rule strengthened, not changed.** Reason upgrades from single-barrel ("breaks pgx") to double-barrel ("redundant *and* breaks pgx") once the in-process-equivalence framing is explicit.

### Details

**The missing link: `pgxpool` is a transaction pooler.** The 0.1.0 entry justified Session Pooling with "prepared statements + IPv4 + full PG feature set" and dismissed Transaction Pooling as "wrong shape for a long-lived Go server." True, but incomplete — it didn't name *why* transaction pooling was wrong-shape, which is: `pgxpool.QueryContext` already leases a backend for the single query's duration and returns it immediately; a `BeginTx` block leases for the transaction's duration. That is transaction-mode pooling semantics, implemented inside the Go process. Supavisor in transaction mode would sit on the wire and do the same job, one layer up and across a network hop, while breaking pgx's prepared-statement cache. Naming the in-process equivalence makes the architectural fit between Go + Direct Connection *positive* instead of framing "we picked Session Pooler" as a compromise. *Where:* new paragraph in `CLAUDE.md` §Database connection; new subsection in `stack.md` §8 ("Database URLs — both Direct Connection, split by role"). *Why:* during a stack review the "FastAPI + PgBouncer transaction mode" pattern was raised as the canonical SaaS default and the implicit challenge was "why aren't we doing that?". The honest answer isn't "we're a special-case orchestrator" (we aren't — v1 Corellia is closer to CRUD-with-two-side-trips-to-Fly than a true 24/7 orchestrator), it's "our driver is doing transaction pooling for us, so the external pooler would be redundant" — a cleaner, more durable framing that survives re-examination.

**Connection count ceiling stated as `instances × pool_size`, not user count.** The direct corollary of the in-process-pooling framing is that Postgres backend count scales with horizontal deployment shape, not traffic. For a single Go binary at `MaxConns=10`, 100 concurrent admin requests use 10 backends held for ~5ms each; they do not use 100. Commercial-scale Corellia (10k orgs) at ~2–5 instances holds 20–50 backends — well under Postgres's ~500 ceiling. The inflection where we'd need Supavisor *session* mode on top is ~50 backend instances, far past v1/v2. Documented explicitly in the new `CLAUDE.md` block and `stack.md` §8 paragraph so future readers don't conflate "high user count" with "connection pressure." *Where:* `CLAUDE.md` §Database connection "Ceiling math" paragraph; `stack.md` §8 new subsection. *Why:* the "scales with users" intuition is load-bearing in the industry-default transaction-pooling argument (Rails/Django/FastAPI shops use it because each request holds a connection for its whole lifetime — a framework-level constraint we don't share). Naming the actual scaling axis (instances, not users) closes the conceptual loop and makes the "when would we add a pooler?" trigger concrete.

**Session Pooler reframed as IPv4 fallback.** Previously the primary recommendation; now named as the specific mitigation for local-dev networks that can't reach the Direct Connection host over IPv6 (Supabase removed IPv4 from the Direct endpoint on the free tier in Jan 2024 — this is also why the Session Pooler exists as a product). Swap is drop-in: same driver, same `pgxpool`, different URL — so per-developer variation costs nothing and doesn't leak into committed config. *Where:* `.env.example` `DATABASE_URL` comment; `CLAUDE.md` §Database connection; `stack.md` §8; `config.go` godoc; `codegen-cheatsheet.md` quick rule. *Why:* the Session Pooler option was overbilled as the "safe default" when it's really just a portability escape hatch. Making the decision tree explicit — try Direct first, fall back only if your network forces it — clarifies the recommendation operationally.

**`.env.example` DATABASE_URL comment rewritten; example URL flipped to Direct.** The committed example now shows `postgres://postgres:<password>@db.<project-ref>.supabase.co:5432/postgres?sslmode=require` (matching what the Supabase dashboard's "Direct connection" tab copies out), with the Session Pooler form retained as an IPv4-fallback paragraph further down the block. `DATABASE_URL_DIRECT`'s example is unchanged (same URL; always was Direct). *Where:* `.env.example` lines 13–30 (replaces the prior SP-primary block). *Why:* the `.env.example` is the first thing a new contributor copies; showing the Direct URL as the canonical example makes the recommendation operational, not just documentary.

**`config.go` godoc rewritten.** Three-paragraph structure preserved (what the var is, what `DATABASE_URL_DIRECT`'s role is, a practical caveat). Content flipped to name Direct + `pgxpool`'s role + the IPv4 fallback + the Transaction Pooler red line. *Where:* `backend/internal/config/config.go` lines 13–26. *Why:* this is the doc comment a developer reads via IDE hover; keeping it aligned with the canonical Markdown explanation avoids drift.

**CLAUDE.md §Migrations heading cleaned up.** Was "always use DATABASE_URL_DIRECT, not the pooler URL"; now "always via DATABASE_URL_DIRECT." The "not the pooler URL" framing was meaningful under the old convention (pooler-vs-direct was a user-facing choice); under the new convention both URLs are Direct and the split is by *role*, so the warning no longer parses correctly. *Where:* `CLAUDE.md` §Common commands §Migrations subheading. *Why:* the old wording would have misled a reader into thinking there was a pooler URL in `.env` to accidentally mistype.

### Behavior change (known)

None at runtime. No Go code was touched except the godoc comment on `DatabaseURL`. The Go binary reads whatever `DATABASE_URL` contains — Direct or Session Pooler — and `pgxpool` opens a pool against it either way. The change is documentary and conventional (what we tell a new operator to paste into `backend/.env`, and what the godoc + `.env.example` show as the canonical example). Migrations continue to use `DATABASE_URL_DIRECT`; the superuser role distinction that keeps `DATABASE_URL_DIRECT` out of `config.Config` is unchanged. The pending local bring-up is unblocked by this (populating `DATABASE_URL` with the Direct URL is now the recommended path), not made harder.

### Supersedes

- **0.1.0 "Two-URL DB strategy."** The line "session pooling gives IPv4 support + multiplexing + full PG feature set (prepared statements, advisory locks)" is historically accurate only for the rejected *Transaction* Pooler — Session Pooler was never actually needed for multiplexing (`pgxpool` does that) or prepared statements (Direct keeps them too). The rationale upgrade lands here.
- **0.2.0 pending-work item** naming "session-pooler `DATABASE_URL`" as a bring-up prerequisite — updated in-place in `docs/completions/frontend-scaffold-completion.md` to read "Direct Connection `DATABASE_URL`."

### Known pending work

- **Local bring-up is still the next blocker** (carried forward from 0.2.1/0.2.2/0.2.3). With the canonical URL now Direct, verify IPv6 reachability from the local network as part of first bring-up — a quick `dig AAAA db.<ref>.supabase.co +short` returning an IPv6 address, plus a one-shot `psql "$DATABASE_URL" -c 'select 1'` smoke test, is enough. ## 0.2.1 — Seeding Removed (2026-04-24)

Removed all seeding from the backend: the default-org `INSERT` in the initial migration, the `defaultOrgID` constant and auto-provisioning branch in `users/service.go`, and the now-unused `CreateUser` query + its sqlc-generated artifacts. Supersedes 0.1.0's "seeds a default org + auto-provisions users on first login" behavior.

### Index

### Details

**Seeding removed from migration.** Deleted the two-line `INSERT INTO organizations (id, name) VALUES ('00000000-0000-0000-0000-000000000001', 'Default Org');` and its comment from `backend/migrations/20260424120000_initial_schema.sql`. The `organizations` and `users` tables remain; only the seed row is gone. *Where:* `backend/migrations/20260424120000_initial_schema.sql`. *Why:* seeded rows are policy smuggled into schema. Provisioning the first org (and the first admin user) is a product decision that should live in an explicit bootstrap flow (admin invite, CLI, or a distinct policy migration), not a silent `INSERT` in the schema migration every dev applies.

**Auto-provisioning removed from domain service.** `users.Service.GetCurrentUser` no longer silently creates a `users` row when `GetUserByAuthID` returns no rows — the sqlc error bubbles up as-is. `defaultOrgID = uuid.MustParse("00000000-0000-0000-0000-000000000001")` constant removed; `github.com/google/uuid` import dropped (was only used by the constant). *Where:* `backend/internal/users/service.go`. *Why:* same rationale as the migration — provisioning is an admin decision, not a side effect of first login. A valid JWT with no matching `users` row should fail visibly rather than auto-granting membership to the seeded default org, which amounted to "anyone who signs in becomes an admin."

**`CreateUser` query removed.** The `-- name: CreateUser :one ...` block removed from `backend/queries/users.sql`; `sqlc generate` regenerated `backend/internal/db/users.sql.go` without `CreateUser` or `CreateUserParams`. *Where:* `backend/queries/users.sql`, `backend/internal/db/users.sql.go`. *Why:* auto-provisioning was the only caller. Per CLAUDE.md's "if you are certain that something is unused, delete it completely" rule, leaving `CreateUser` in place as a latent helper would be premature — when an actual provisioning flow arrives (admin invite, bootstrap migration, etc.), it'll define the INSERT it actually needs, which may or may not match today's shape.

### Behavior change (known)

A valid-JWT-but-no-`users`-row request to `GetCurrentUser` now returns `pgx.ErrNoRows` through the domain service. The handler (`httpsrv/users_handler.go`) currently wraps all service errors as `connect.CodeUnauthenticated`, which is the wrong Connect code for "authenticated but not provisioned" — the right code is `CodePermissionDenied` or `CodeFailedPrecondition`. Effect on the FE: the dashboard's `useEffect` catch branch displays the raw error, and the root `/` redirect still routes authenticated users to `/dashboard`, so the user sees an error page rather than an infinite sign-in loop. Not fixed in this pass to keep scope tight; tracked below.

### Known pending work (added by 0.2.1)

- **Error mapping in `users_handler.go`.** Translate `pgx.ErrNoRows` (and a service-layer sentinel like `ErrUserNotProvisioned`) to an appropriate Connect code (likely `PermissionDenied`), so the FE can render a meaningful "your account isn't provisioned — contact an admin" state instead of a generic error string.
- **First-admin bootstrap path.** Replace what the seeded default-org + auto-provisioning used to do. Options: an admin-invite flow that creates `users` rows explicitly, a separate policy migration that seeds a single bootstrap admin from an env var, or a one-shot CLI command. Pick one when the admin UX in blueprint §10 is specified.
